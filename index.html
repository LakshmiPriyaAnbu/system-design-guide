Beginner-Friendly Handbook

System Design, explained like you're new.

Every concept in plain English. Every jargon term defined inline the first time it appears. Bullets for the fundamentals so you can scan; the case studies at the end are written as short articles you can read like a book chapter.

How to read this
Skim the sidebar. Any word in blue-bold is a term worth learning by name. If something in the case studies mentions a term you don't know, it's almost certainly explained in the earlier bullet sections.
Part 1 — Foundations

01What is system design?

  • System design = planning how software will work at scale, before writing the code.
  • Given a vague prompt like "design Instagram," you plan:
    • What the system does (its features).
    • How well it must do it — users, speed, reliability (called non-functional requirements).
    • What pieces exist: clients, servers, databases, caches, queues.
    • How the pieces talk to each other.
    • What happens when something breaks.

The 5-step framework — memorize this

  1. CLARIFY — ask 4–6 questions to pin down scale and needs.
  2. ESTIMATE — rough math (users × requests per user = QPS).
  3. HIGH-LEVEL — draw major boxes and arrows.
  4. DEEP-DIVE — pick 1–2 components and go deep.
  5. SCALE & FAIL — what if it grows 10×? What if X dies?

02Client & server

  • Client = anything that asks for data (your phone, browser, IoT device, another server).
  • Server = a program that waits for requests and sends responses.
  • They talk over a network using a shared protocol — most often HTTP.
Client ──── request ────►  Server
       ◄─── response ────

03HTTP, REST & APIs

HTTP in one sentence

  • HTTP (HyperText Transfer Protocol) = "the language browsers and servers speak."
  • Client sends one request. Server sends one response. That's it.

HTTP methods (verbs)

MethodMeaningIdempotent?
GETRead dataYes
POSTCreate something newNo
PUTReplace the whole thingYes
PATCHUpdate part of itUsually no
DELETERemoveYes
  • Idempotent = doing it twice has the same effect as doing it once. If the network hiccups and your app retries, you don't want the user charged twice.

Status codes — the number the server sends back

  • 2xx = success: 200 OK, 201 Created, 204 No Content.
  • 3xx = redirect: 301 Moved Permanently.
  • 4xx = your fault: 400 bad, 401 not logged in, 403 not allowed, 404 not found, 409 conflict, 429 too many.
  • 5xx = server's fault: 500, 502, 503, 504.

REST — a style for designing HTTP APIs

  • Use nouns in URLs (/users), not verbs.
  • Use HTTP methods for the action.
  • Version your API: /v1/users.
  • Paginate long lists: ?limit=20&cursor=abc.

WebSocket — when HTTP isn't enough

  • HTTP is one-way — only the client can start a conversation.
  • If the server needs to push ("new message!"), plain HTTP is awkward.
  • WebSocket = a persistent two-way pipe between client and server. Used in chat, live dashboards, games, IoT.

04DNS

  • DNS (Domain Name System) = the "phonebook of the internet."
  • Turns api.instagram.com into an IP address like 52.84.123.45.
  • Cached at many layers (OS, browser, resolver) so it's not looked up every request.
  • GeoDNS returns a different IP based on your location — the basis of CDN routing.

05Load balancers

  • One server handles ~1,000 req/s. You need 100,000 req/s → ~100 servers. Clients only know one address. Who splits the work?
  • Load balancer (LB) = a box in front of your servers that spreads incoming requests.
                  ┌──────────┐   ┌── Server 1
   Clients ────►  │   Load   │──►│── Server 2
                  │ Balancer │──►│── Server 3
                  └──────────┘   └── ...

How it picks a server

  • Round-robin — 1, 2, 3, 1, 2, 3…
  • Least connections — whoever has the fewest active users.
  • Weighted — bigger servers get more.
  • IP hash — same client always → same server.

L4 vs L7

  • Layer 4 — routes by IP + port. Fast, doesn't understand HTTP.
  • Layer 7 — understands HTTP. Can route by URL, headers, cookies. Slower, smarter.

Health checks

  • LB pings each server every few seconds. 3 fails in a row → remove from pool. Recovers → add back.

06API gateway

  • API gateway = a smart front door for your system.
  • Handles: routing, authentication, rate limiting, TLS termination, logging, request transformation.
  • Load balancer vs gateway: LB spreads to identical backends; gateway routes to different backends and adds smarts.
  • Examples: AWS API Gateway, Kong, NGINX, Envoy.

07SQL vs NoSQL databases

SQL (relational)

  • Data in tables with rows and columns.
  • Tables link via foreign keys.
  • Query language: SQL.
  • Strengths: transactions (ACID), joins, mature.
  • Weakness: hard to split across many machines (sharding).
  • Examples: PostgreSQL, MySQL, Oracle.

NoSQL — four flavors

  • Key-value — a giant hash map. Redis, DynamoDB.
  • Document — JSON blobs. MongoDB, Firestore.
  • Wide-column — rows with sparse columns; built for huge writes. Cassandra, HBase.
  • Graph — nodes + edges. Neo4j.
  • Time-series (special case) — sensor/metric data. InfluxDB, TimescaleDB.

08Indexes, transactions, ACID, BASE

Index

  • Like a book's back-index — lets the DB jump straight to matching rows without scanning everything.
  • Trade-off: faster reads, slower writes (every insert also updates the index).

ACID — the four transaction guarantees

  • Atomicity — all steps happen or none do.
  • Consistency — the DB moves from one valid state to another.
  • Isolation — concurrent transactions don't see each other's half-done work.
  • Durability — once committed, data survives crashes.

BASE — the NoSQL counterpart

  • Basically Available, Soft state, Eventual consistency.

09Caching

  • Cache = a small, fast copy of data kept close to whoever's reading.
  • RAM is ~100× faster than a DB round trip. Caches make apps 10–1,000× faster.

Where caches live

  • Browser → CDN edge → API gateway → in-process → distributed (Redis/Memcached) → DB → disk.

Write strategies

  • Write-through — write cache + DB together. Fresh, slow.
  • Write-behind — write cache, ack user, flush DB later. Fast, risky.
  • Write-around — write only to DB; cache fills on read miss.

Read strategy: cache-aside

  1. Look in cache.
  2. If not there, read from DB.
  3. Store in cache with a TTL.

Eviction — kick things out when cache is full

  • LRU — least recently used (most common).
  • LFU — least frequently used.
  • TTL — expire after N seconds.

Cache pitfalls

  • Stale data — cache old copy. Fix: TTL or invalidate on write.
  • Thundering herd — hot key expires, 1,000 users hit DB. Fix: single-flight lock.
  • Cache penetration — repeated lookups of nonexistent keys. Fix: cache "not found" too.
  • Cache avalanche — many keys expire together. Fix: add jitter.
  • Hot key — celebrity profile crushes one node. Fix: replicate.

10Message queues & events

  • Service A calls B directly = coupled. B slow → A slow. B down → A breaks.
  • Message queue = a buffer that holds "please do this later" messages.
  • A drops the message and moves on. B reads when ready. If B is down, messages pile up safely.

Queue vs pub/sub

  • Queue — each message goes to one consumer. AWS SQS, RabbitMQ.
  • Pub/sub — each message goes to every subscriber. Kafka, SNS.

Kafka

  • Not really a queue — a distributed log.
  • Topics split into partitions (ordered streams). Messages kept for days/weeks.
  • Different consumer groups read at their own pace.

Delivery guarantees

  • At-most-once — may lose messages.
  • At-least-once — may duplicate. Common; make consumers idempotent.
  • Exactly-once — hard.

11CDN

  • Network of edge servers around the world caching static content close to users.
  • Mumbai user → Mumbai edge in ~5ms instead of US origin in ~200ms.
  • What goes in: images, video, JS, CSS, fonts, firmware.
  • Invalidation via TTL, purge, or cache-busting filename (app.a3f8b9.js).
  • Examples: CloudFront, Cloudflare, Fastly, Akamai.

12Object storage (S3)

  • An HTTP service that stores blobs (files) accessible by key.
  • Cheap, effectively unlimited, extremely durable.
  • Use for: user uploads, backups, static hosting, firmware images.
  • Handy: versioning, lifecycle policies (auto-tier to cold storage), presigned URLs.

13Rate limiting

  • Cap how many requests a client can make per time window.
  • Algorithms:
    • Fixed window — count per minute. Boundary spike problem.
    • Sliding window log — timestamp per request. Precise, memory-heavy.
    • Sliding window counter — weighted blend. Smooth, cheap.
    • Token bucket — bucket of N tokens, refills at rate R. Allows bursts. Most popular.
    • Leaky bucket — constant-rate drain. Smooth output.
  • Distributed via Redis with atomic operations. Reject with HTTP 429.

14Auth

  • Authentication — who are you? Authorization — what may you do?

Session-based

  • Server issues a session ID, stores it, sets a cookie. Easy to revoke; needs shared state.

Token-based (JWT)

  • JWT = a signed string. Server verifies signature, no DB lookup. Stateless, scales easily.
  • Hard to revoke before expiry — use short access tokens + longer refresh tokens.

OAuth 2.0 — "Log in with Google"

  1. Redirect to Google. User approves.
  2. Google returns a code.
  3. Your backend exchanges code + secret → access token.
  4. Your app never sees the user's Google password.

Authorization models

  • RBAC — roles + permissions.
  • ABAC — rules over attributes.
  • ACL — explicit per-resource grants.

15Scaling

  • Vertical — bigger box. Easy but capped.
  • Horizontal — more boxes. Practically unlimited.
  • Prefer stateless app servers so any request can go to any server.

Database scaling

  • Replication — one primary handles writes, replicas handle reads. Replicas lag ms–seconds behind.
  • Sharding — split data across DBs by a key.
    • Range sharding — simple but hot ranges.
    • Hash sharding — uniform but resharding is painful.
    • Consistent hashing — adding a node only moves ~1/N of keys.
  • Never shard by a monotonically-increasing key (like created_at) — creates one hot shard.

16CAP theorem & consistency

  • Under a network partition, you must choose Consistency or Availability. Since networks fail, choose CP or AP.
  • MongoDB defaults to CP. Cassandra, DynamoDB are AP.
  • PACELC: if Partition, choose A or C. Else, choose Latency or Consistency.

17Observability

  • Logs — individual events. "What happened at 3:04?"
  • Metrics — numbers over time. "What's our request rate?"
  • Traces — a single request's journey across services.
  • Alerts on p99 latency > 500ms, error rate > 1%, etc.
Part 2 — Classic Problems

18URL shortener

  • Long URL → short URL (tinyurl.com/aB3xZ9). Redirect back on click.
  • Assume 100M writes/day, 10× reads → design for ~12,000 reads/sec.
  • Generate short codes with a counter + base62 encoding. Pre-allocate ID batches to each app server so you don't fight over one counter.
  • Store in a sharded DB, cache hot URLs in Redis.
  • On click: cache hit → 302 redirect. Cache miss → look up DB, cache it, redirect.
  • Analytics: don't write to DB per click — publish to Kafka, aggregate hourly.

19Chat app

  • Phones open a WebSocket to a chat gateway (long-lived pipe).
  • Presence service (Redis) maps user_id → which gateway they're on.
  • Send flow: Alice's gateway persists message + acks her → looks up Bob's gateway → forwards → Bob's gateway pushes to Bob's WebSocket.
  • Store messages in Cassandra: partition by conversation_id, sorted newest-first.
  • Offline: save + push via APNs/FCM. When Bob comes back, pull messages since his last-seen pointer.
  • Groups: don't fan out to each inbox — store once per conversation with per-user "last read" pointers.

20Notification system

  • Producer publishes event → Notification API → Preference service (channel opt-ins) → Template service (render) → Kafka topic per channel → workers for APNs / SES / Twilio.
  • Each notification has an idempotency ID so retries don't duplicate.
  • Retry with exponential backoff. Cap attempts.
  • Priority lanes: OTPs get a fast lane; marketing goes in a slow lane.
  • Big marketing blast: spread over ~5 min to avoid slamming APNs.

21Ride booking (Uber)

  • Location index answers "drivers within R km of point P" in milliseconds. Options: geohash, quadtree, Google S2, Redis GEO.
  • Match: query nearby → score by distance × ETA × rating × surge → offer to top driver with 10s timeout.
  • Driver heartbeat every 4s writes location to Redis with TTL — stale drivers vanish automatically.
  • Trip state machine: REQUESTED → MATCHING → ACCEPTED → ARRIVED → IN_PROGRESS → COMPLETED. Each transition emits a Kafka event.

22Instagram feed

  • Two extremes: pull at read time (cheap writes, slow reads) or push at write time (fast reads, celebrity meltdown).
  • Hybrid: push for normal users; celebrities are marked and merged in at read time.
  • Posts in Cassandra. Media in S3 + CDN. Feed cache in Redis (per-user list of post IDs).

23Dropbox / Google Drive

  • Chunking: split files into ~4MB pieces. Upload in parallel; resumable.
  • Deduplication: hash each chunk; if the server already has that hash, don't re-upload.
  • Sync: each device holds (file_id, version); pull the delta on wake-up. Conflicts: last-write-wins or create "conflicted copy."

24Video streaming (YouTube / Netflix)

  • Upload → transcode to many bitrates → chop into ~6-second HLS segments + manifest → push to CDN.
  • Adaptive bitrate: the player measures your bandwidth every few seconds and picks a bitrate per segment.
  • Popular videos live on CDN edges; the origin is rarely hit.

25Payments

  • Non-negotiables:
    • Idempotency — client sends Idempotency-Key; duplicate → same response.
    • Atomicity — debit + ledger in one DB transaction.
    • Audit log — every state change appended.
    • Reconciliation — nightly match against bank statements.
  • Double-entry ledger: every transaction is two rows that sum to zero.
  • Never store raw card numbers — send to the payment gateway, store their token.

26Search autocomplete

  • Data structure: a trie. Each node holds a letter; walking the tree spells words.
  • At each node, pre-compute the top-K most popular completions.
  • Each app server holds the whole trie (~500MB) in memory.
  • Nightly pipeline rebuilds the trie from search logs.

27Distributed rate limiter

  • Counter in Redis, updated atomically via a Lua script (token bucket refill + take).
  • Each gateway can hold a local bucket and reconcile with Redis every 100ms — fewer Redis calls, tiny over-allowance.
  • Redis down → fail open with a local cap. Better to allow than block everything.
Part 3 — IoT & Mobile

28Mobile app backend

  • Different from web: flaky networks, battery matters, push instead of polling, old app versions live forever.
  • Version APIs (/v1, /v2).
  • Feature flags = server toggle to disable a broken feature without a new app.
  • Force-update endpoint tells old clients "please upgrade."
  • Delta sync: client sends last sync token; server returns only changes.

29Multi-device sync

  • Conflict strategies:
    • Last-write-wins — highest timestamp wins.
    • CRDT — data types that merge in any order (Figma, Linear, Apple Notes).
    • Operational Transformation — server transforms concurrent ops (Google Docs).
  • Every item has a soft-delete flag; hard-delete only after tombstone TTL.
  • Silent push wakes the app to sync — better than polling.

30Offline-first

  1. Local DB is the UI's source of truth.
  2. Writes go to local DB + an outbox queue.
  3. Outbox survives restarts.
  4. UI shows optimistic state; reconciles on server ack.

31Push notifications (deep)

  • iOS: APNs. Server has a .p8 key; device provides a token; server sends via HTTP/2.
  • Android: FCM. Same shape.
  • Payload limited to 4KB.
  • Silent push (iOS: content-available: 1) wakes the app briefly for background sync.
  • Token dies → delete it (APNs returns 410 Gone).

32BLE / Wi-Fi device setup

  1. Device boots into setup mode, broadcasts BLE.
  2. Phone app scans, connects (encrypted).
  3. App reads visible SSIDs, user picks + enters password.
  4. App sends creds over BLE.
  5. Device joins Wi-Fi, calls cloud to claim itself.
  • Security: no plaintext creds over BLE; use ECDH or a static device key from a QR code. Every device has a factory cert. Nonces prevent replay.

33IoT telemetry

  • Devices publish via MQTT — a tiny protocol built for unreliable networks.
  • MQTT broker → IoT gateway (auth, dedup) → Kafka → stream processor / time-series DB / S3.
  • MQTT features: broker buffers messages while device is offline; QoS 0/1/2; tiny headers.

34Time-series storage

  • Optimized for append-only writes and time-range queries.
  • Chunks by time, columnar storage, delta / delta-of-delta encoding, retention policies.
  • Avoid: heavy updates, or user_id as a tag (cardinality explodes).

35OTA firmware updates

  • Two firmware slots (A/B partitions). Current runs from A; new is written to B. Boot pointer swaps.
  • If new firmware doesn't check in cleanly, bootloader flips back to A. Can never brick.
  • Staged rollout: bucket by hash(serial) % 100; server "only if bucket < 10" = 10% rollout.
  • Every image signed by vendor key; device verifies before flashing.
Part 4 — Low-Level Design (LLD)

36OOP essentials

  • An object bundles state (fields) with behavior (methods).
  • Four pillars:
    • Encapsulation — hide internals.
    • Abstraction — expose what, hide how.
    • Inheritance — one class extends another (use sparingly).
    • Polymorphism — same method name, different behavior per subclass.

37SOLID

  • Single Responsibility — one reason to change per class.
  • Open/Closed — open to extend, closed to modify.
  • Liskov — subclasses work anywhere the parent does.
  • Interface Segregation — small focused interfaces, not big fat ones.
  • Dependency Inversion — depend on abstractions, inject them in.

38Design patterns worth knowing

  • Singleton — one instance (use rarely).
  • Factory — one method returns different subtypes.
  • Builder — step-by-step construction.
  • Adapter — make incompatible interfaces fit.
  • Decorator — wrap to add behavior.
  • Facade — one simple front door.
  • Composite — trees where leaves and containers share an interface (folders/files).
  • Observer — subjects notify subscribers.
  • Strategy — swap out algorithms via an interface.
  • State — behavior changes with internal state; replaces big if/else.
  • Command — action as an object (undo, queue).

39Thread safety

  • Race condition — timing-dependent bug from concurrent access.
  • Mutex — only one thread inside the critical section.
  • Semaphore — up to N threads.
  • Atomic — indivisible operation.
  • Deadlock fix: acquire locks in the same order everywhere.
  • Prefer immutable data — no locks needed.

40LLD problem sketches

  • Parking Lot — Lot → Floor → Slot; Vehicle hierarchy; FeeCalculator (Strategy).
  • Elevator — Scheduler as Strategy (LOOK algorithm).
  • Splitwise — SplitStrategy (Equal/Exact/Percentage); balances form a directed graph.
  • Chess — Piece hierarchy with validMoves; Move as Command → undo.
  • ATM — State pattern (Idle → CardInserted → Authenticated → Transacting).
  • Logger — Composite of Appenders; Async logger as Decorator.
  • File System — Composite (Directory + File). size() recurses.
  • Vending Machine — State pattern.
  • Library — separate Book (title) from BookCopy (physical instance).
Part 5 — Interview Execution

41The 45-min system-design playbook

  • 0:00–0:05 Clarify — read/write mix, DAU, latency target, consistency needs.
  • 0:05–0:10 Estimate — QPS, storage, bandwidth. State assumptions.
  • 0:10–0:20 High-level — draw clients, gateway, services, DB, cache, queue. Trace one main use case.
  • 0:20–0:35 Deep-dive — 1–2 components. Schema, sharding, indexes, a concrete algorithm, a failure scenario.
  • 0:35–0:42 Scale & fail — 10× load, DB death, region outage.
  • 0:42–0:45 Wrap — recap trade-offs, 1–2 improvements.

42The 45-min LLD playbook

  • Clarify. List nouns → classes. Sketch a class diagram.
  • Code the 2–3 interesting methods for real.
  • Show how adding feature X fits without rewrites.
  • Patterns that answer 80% of LLD prompts: Strategy, State, Observer, Factory, Composite, Decorator, Command.

43Behavioral (STAR)

  • Situation → Task → Action → Result.
  • Prep 4–6 stories: simplified a complex thing, owned an outcome, disagreed with a senior, delivered on a deadline, recovered from a mistake, mentored someone.
  • Use specific numbers.

44Common rejection reasons

  • Jumping into design before clarifying.
  • 15 boxes, deep on zero.
  • No trade-offs.
  • Hand-waving the hard part.
  • Ignoring failures.
  • Silence.
Part 6 — Case Studies (article style)

Each case study below is written as a short article — the way a beginner textbook would explain the system. Every technical term is defined the first time it appears. If you want to skim, read the first paragraph; if you want depth, read the whole thing.

50How WhatsApp works

WhatsApp handles roughly two billion users and moves more than a hundred billion messages every single day. At that scale, ordinary design choices break down; the whole system is shaped around one goal — keeping billions of people connected all at once without needing an army of servers.

The most surprising choice is the language WhatsApp is built in: Erlang. Erlang is an old language, invented at Ericsson in the 1980s to run telephone switches. Its special talent is holding an enormous number of tiny concurrent tasks at the same time on one machine. A typical Java or Python server can keep about a hundred thousand people connected before running out of memory; a WhatsApp server built on Erlang holds about two million. Twenty times fewer servers means dramatically lower cost and simpler operations.

Your phone doesn't send an HTTP request every time you type. Instead, when you open the app, it opens one long-lived TCP connection to a WhatsApp gateway and keeps that "pipe" open for as long as possible. Messages travel through that pipe using a custom protocol inspired by an older chat standard called XMPP.

Now the important part: WhatsApp cannot read your messages. This is end-to-end encryption (often shortened to E2E), and WhatsApp uses the same math the Signal app is famous for. When you first start a conversation with someone, your phone and theirs perform a mathematical dance called X3DH that lets both sides agree on a secret key without ever sending the secret over the network. After that, each message uses a slightly different key thanks to a technique called the Double Ratchet — even if a bad actor stole one key, they could only decrypt one message. The WhatsApp server never sees the secret; from its point of view, every message is a scrambled blob. It only sees who sent something to whom, and when. This distinction — content is invisible, metadata is visible — is important to understand about all E2E systems.

Because most users now have several devices (phone, tablet, laptop), each device has its own key pair. When a friend sends you a message, their phone encrypts it separately for each of your devices. Your phone remains the "primary" — new devices are added by scanning a QR code with the primary.

Voice and video calls don't use WhatsApp servers directly. They use WebRTC, the standard peer-to-peer protocol that browsers and apps use to stream audio and video. When two phones can reach each other directly, calls flow between them. When they can't — often because they're behind home routers that hide their true addresses, a problem called NAT — WhatsApp provides a middleman server called a TURN relay that both phones can reach, and the video flows through it.

51How Twitter builds your timeline

The core problem Twitter must solve is deceptively simple: when you open the app, how does it show you posts from every account you follow, quickly? For a normal user with a few hundred follows, this is easy. For a system with hundreds of millions of users, some of whom follow thousands of accounts and some of whom have hundreds of millions of followers, it becomes a genuinely hard problem.

Twitter's original approach was called fan-out on write. The idea is: do the work at the moment someone posts, not when someone reads. When you tweet, a background worker looks up your followers and copies your new tweet's ID into every one of their personal "inbox" lists. These lists live in Redis, a very fast in-memory database. When one of your followers opens the app, the server runs one Redis command (called LRANGE) to grab the most recent fifty tweet IDs from their inbox, then hydrates them — a term that just means "look up the full tweet content by ID and attach it."

This works beautifully for the vast majority of users. Where it breaks is celebrities. If Cristiano Ronaldo tweets and he has six hundred million followers, the naive approach means writing six hundred million entries into six hundred million Redis lists for a single tweet. That's expensive; it takes forever and stresses the entire system.

The fix is a hybrid. Normal users still get fan-out on write; it's cheap for them. Celebrities are marked as such and their tweets are not fanned out. When any of their followers opens the app, the system separately fetches recent celebrity posts and merges them into the timeline at read time. This mixes the two approaches so neither one dominates.

The tweets themselves are stored in Twitter's own key-value database called Manhattan. Every tweet has an ID from a system called Snowflake. A Snowflake ID is a 64-bit number split into three parts: about 41 bits of timestamp (milliseconds since 2010), 10 bits identifying the machine that generated it, and 12 bits of counter for tweets generated in the same millisecond. Because the timestamp sits in the highest bits, sorting Snowflake IDs numerically automatically sorts them by time — no separate ORDER BY created_at query needed. It's a small trick with a huge downstream benefit.

52How YouTube works

YouTube is really two systems joined at the hip: a system for accepting and processing videos from uploaders, and a system for streaming those videos to viewers. Both are enormous, but their engineering challenges are very different.

Metadata — video titles, descriptions, channel information, watch history — lives in an underlying database that runs on top of MySQL. YouTube uses a project called Vitess to shard MySQL across many machines, giving them the reliability of a familiar SQL database with the scale of a distributed system. Views, likes, and other counters live in Bigtable, which is optimized for very high write throughput. Videos themselves are enormous — hundreds of petabytes — so they sit in a specialized cold blob storage system rather than any conventional database.

When you press play, the actual bytes come from a CDN (Content Delivery Network) — a global network of cache servers close to viewers. The video isn't a single file but a series of ~6-second chunks in a format called HLS or DASH, along with a manifest file that tells the player what chunks are available at what quality. The player measures your bandwidth as you watch. If your Wi-Fi is strong, it downloads the next chunk in 1080p. If your bandwidth drops, the next chunk comes in 480p. This "pick a bitrate per chunk" behavior is called adaptive bitrate streaming, and it's why YouTube feels smooth even on flaky networks.

The famous "Up Next" recommendation panel uses a machine learning model called a two-tower model. Imagine two big lookup tables. One tower turns each video into a vector — a list of numbers representing that video's "meaning" (built from the title, captions, and how people watched it). The other tower turns each user into a similar vector — representing that user's tastes. To find candidates for you, the system finds videos whose vector is close to yours (a math operation called cosine similarity). This gives ~500 candidates; a second, deeper model then re-ranks them into the exact order you see. Diversity is layered on so you don't see ten videos from the same creator.

Live streaming is a separate pipeline. A creator's software sends video via RTMP (an older live-streaming protocol) to YouTube's ingest servers, which transcode on the fly, chop the stream into HLS segments, and push those segments to the CDN. From click to viewer's screen is typically 5 to 30 seconds of latency, with a special "low-latency HLS" mode getting down to ~3 seconds.

53How Uber computes your ETA

When you request a ride, one of the numbers on your screen is your estimated time of arrival. That number seems trivial. It isn't. Uber computes it at ferocious scale — potentially a million times a second globally — and it must be under 100 milliseconds every time, every place, no exceptions. Getting there requires three stages that work together.

Stage one is map matching. Your phone reports a raw (latitude, longitude) coordinate that's usually slightly off — a few meters north of the actual road, perhaps. The system snaps that coordinate to the nearest road segment on a precomputed road graph. This gives everyone downstream a clean, canonical position.

Stage two is routing — finding the shortest path from origin to destination on the road graph. This is the classic shortest-path problem, but you can't afford to run a plain Dijkstra algorithm on a graph with millions of edges under a strict latency budget. Uber uses a technique called Contraction Hierarchies, which pre-processes the road graph into a form that allows lookups thousands of times faster than the naive algorithm.

Stage three is the interesting one: ETA prediction. Pure shortest-path routing gives you a distance-based guess, but reality is messier. Traffic exists. Signals turn red. It rains. Uber trains a machine learning model — first a gradient-boosted tree, and more recently a transformer called DeepETA — that ingests time of day, day of week, recent traffic on each segment (from other drivers phones), historical traffic patterns for that time and place, weather, local events, and even the driver's current speed. It outputs a predicted number of seconds. The magic is that this model isn't retrieving a stored answer; it's producing a fresh number for every request in a few milliseconds.

The traffic data itself flows in real time: driver phones stream their speed and location into a stream processor (Uber uses Flink for this), which aggregates the numbers per road segment and feeds them into a feature store — a very fast key-value database of "what's happening on each segment right now" that the ETA model reads at request time.

54How Uber finds nearby drivers at 1M req/s

Every time you open the Uber app and see little cars around you, the system just answered a geographic question: which drivers are within some radius of this point? Multiply that by every rider on Earth opening the app simultaneously and you have one of the highest-throughput geographic search problems anywhere.

The naive approach would be to index driver locations in a spatial database like PostGIS. That works up to a point. At Uber's scale, it collapses. So they invented something better, built on a clever piece of math from Google called S2.

S2 divides the surface of the Earth into a grid of cells. The grid is hierarchical — you can zoom in from large cells (a whole continent) down to tiny ones (a city block). Each cell has a 64-bit ID. Here's the trick: the IDs are ordered along a special curve called a Hilbert curve, arranged so that cells whose IDs are numerically close are geographically close on the Earth's surface. That property is powerful; it means "find all cells near me" becomes "find all IDs in this range."

Uber shards the world by S2 cell ID range: shard 1 owns cells in one range, shard 2 owns another, and so on. Driver locations live in RAM (not a database) on the shard that owns their current cell. Drivers send a heartbeat every 4 seconds updating their position. When you ask "who's nearby?", the query goes to the small handful of shards whose cell IDs cover your search radius. Each returns the drivers it knows about, and the results merge.

The whole system is designed so writes and reads flow through different paths. Writes (driver heartbeats) go by driver ID to a "home" shard; a separate propagation keeps the cell index up to date. Reads (find-nearby) hit the cell index. This decoupling is what lets the system serve a million queries a second at p99 latency under 10 milliseconds.

55How Airbnb works

Airbnb feels like a search engine glued to a booking system, and that's exactly what it is. The search side has to handle open-ended queries — location, dates, price, amenities, guest count — with personalization on top. The booking side has to do something search engines don't: guarantee that no two people can book the same night in the same home.

Search runs on ElasticSearch, which is designed for exactly this kind of "filter by many fields, rank by relevance" workload. Each listing is indexed with its geographic point, price, amenities, and — this is the tricky part — a pre-computed availability calendar covering the next year. Yes, that means storing a row per listing per night; the numbers work out because that's still tiny compared to the search work itself.

Booking is where the hard consistency lives. Imagine two families in different countries clicking "book" for the same beach house for the same weekend at the same instant. Both requests reach the server. Only one should win. Airbnb solves this by treating each night as its own database row with a status: available, held, or booked. A booking request opens a database transaction that atomically flips every night in the range from available to held. If any of those nights isn't available, the whole transaction fails and rolls back. The winning request now has a 15-minute hold while payment processes. On payment success, held flips to booked. On payment failure, the nights revert to available. Two concurrent bookings can't both succeed because the database will let only one of them flip those rows.

Pricing is more layered than it looks. Hosts set a base price; they can add per-night overrides (weekends, holidays); Airbnb's smart-pricing model suggests dynamic adjustments; then platform fees and taxes are computed on top — all at booking time so the final number matches what the guest saw. Trust and safety runs a machine-learning risk score on every booking; suspicious patterns land in a manual-review queue.

56How Reddit works

Reddit's engineering shape is set by three problems: how to rank posts, how to store threaded comments at scale, and how to fight bots. Its ranking algorithms have become classics of internet lore, taught in engineering interviews for years.

The formula for the "Hot" feed is worth learning by heart. It's roughly log10(max(|score|, 1)) × sign(score) + seconds_since_epoch / 45000. The logarithm is the interesting part. It means the first 10 upvotes on a post count for one unit of rank, the next 90 (getting to 100) count for another unit, and the next 900 (getting to 1000) count for another. Diminishing returns. This prevents already-popular posts from dominating forever. The time term adds a linear boost for fresh posts; older posts sink naturally as the clock ticks.

Comments are ranked differently. The "Best" algorithm uses something called the Wilson score lower bound, a piece of statistics that gives a "worst-case" estimate of a comment's approval rate given how many people voted. A comment with 8 upvotes and 2 downvotes is more trustworthy than one with 4 upvotes and 0 downvotes even though the raw ratio looks worse — because more data means more confidence.

Threaded comments create their own storage challenge. A viral post can have a hundred thousand replies nested many levels deep. Reddit stores each comment with pointers to its post and its parent comment, indexed for fast retrieval. When you open a post, the app fetches only the top-ranked visible comments; deeper branches load on demand as you scroll. Very hot subreddits have their feeds precomputed by background workers; smaller ones compute their rankings on demand.

The final trick is vote fuzzing. Publicly displayed vote totals are slightly perturbed — a post that really has 1,247 votes might show 1,251. Real totals are stored separately. This tiny amount of noise defeats bots that try to detect whether their upvote "took" and helps prevent manipulation.

57How Google Search works

Between the moment you press Enter and the moment results appear is roughly two hundred milliseconds. In that window, Google does an astonishing amount of work — crawling, indexing, matching, ranking, and rendering — using techniques that took decades to invent. Understanding the pipeline is useful even if you never build a search engine.

Everything starts with crawling. Google's crawlers continuously fetch pages from across the web, respecting rules pages set in a file called robots.txt. New pages are prioritized by signals like PageRank and how often the site updates.

Crawled pages feed into the index. The index is an inverted index — for every word in the vocabulary, a list of the documents that contain that word and where. Think of it as a huge dictionary at the back of every book on the internet combined. The index is split across many shards, each holding a slice of documents. Every single query, no matter how obscure, is sent to every shard in parallel.

When you search, several things happen in quick succession. First, the query is tokenized and spell-corrected. Then it's expanded — "car" might automatically include "automobile" and "vehicle." Then it hits every shard. Each shard intersects the posting lists for your query words, picks its top candidates, and returns them. An aggregator merges all the shard responses into one global top list.

Now comes ranking, where hundreds of signals combine. PageRank (an early Google invention) measures a page's importance based on the link graph. BERT and MUM are language models that evaluate whether the page's text actually answers your query semantically, not just by keyword match. Click models measure which results people historically click. Freshness matters for news queries. Site quality matters ("did people report this site as spam?"). Personalization tunes for your location and history.

The last critical piece is tail-latency control. If one shard out of thousands is temporarily slow, the whole query would be slow — so Google sets a time budget and simply skips laggard shards past that budget. Sometimes it sends the same query to a backup replica after a fraction of a second and takes whichever response comes back first. This is called backup requests and is a defining technique of large distributed systems.

58How Amazon S3 works

S3 is the archetypal object storage service. Amazon claims eleven nines of durability — a probability of about one in a hundred billion of losing an object per year. How they achieve that number involves math that's worth understanding.

Naive replication would store three copies of every file — one primary, two backups — giving triple the storage cost. S3 uses something smarter called erasure coding. When you upload a file, S3 splits it into K data chunks (say 10) and computes M additional parity chunks (say 4). This is math based on a technique called Reed-Solomon codes; the key property is that any 10 of the 14 chunks are enough to reconstruct the original. If 4 chunks are lost, you can still recover the file — and the storage overhead is 1.4× the original, not 3×.

The chunks are spread across racks and across data centers (availability zones) so that a single rack failure, or even a whole data center failure, doesn't cause data loss. When you read an object, S3 reads chunks in parallel and reconstructs; failures during reads are handled by trying other chunks.

The service itself has three main layers. Request routers sit at the front, doing authentication, quota checks, and looking up where objects live. A keymap service acts as an index: given a bucket and key prefix, it tells you which storage node holds the data. The storage nodes are the fleet that actually holds the erasure-coded chunks.

Until 2020, S3 was famously eventually consistent — you could write an object and, for a brief window, read the old version. In 2020 they added strong read-after-write consistency without adding meaningful latency, using a coordinator-based protocol during reads. It was a quiet revolution — a system this large moving to strong consistency without slowing down was widely thought impossible.

59Kafka internals — why it's so fast

Kafka is deceptively simple on the surface. Producers put messages in; consumers take them out. What makes it interesting is how it manages to be dramatically faster than every "obvious" alternative — often by 100× or more. The reasons come down to a few unfashionable techniques that turn out to matter enormously.

The first is that Kafka writes its data to disk as an append-only log. Every partition is a big file, and every new message is written at the end. There's no B-tree to update, no index to rewrite, no random seeks. This matters because sequential writes to a disk — even a mechanical one, but especially an SSD — are dramatically faster than random writes. Hundreds of megabytes per second on cheap hardware.

The second is zero-copy. When a consumer reads messages, Kafka doesn't copy the data into userspace and back out to the socket. It uses a Linux system call called sendfile that lets the kernel move bytes directly from the operating system's page cache (where files are buffered) to the network socket. No intermediate copy, no wasted CPU. Combined with the OS aggressively caching recent messages in RAM, most reads never touch disk at all.

The third is partitioning. Each topic is split into partitions, and consumers can read partitions in parallel. Adding partitions is how you scale a topic to more throughput.

Reliability comes from replication. Each partition has a leader and some number of followers. Producers write to the leader; followers copy from it. The subset of followers that are caught up is called the ISR, or In-Sync Replicas. If the leader dies, one of the ISR members takes over as the new leader — no data loss because ISR members are guaranteed to have everything the leader had.

Consumers keep track of where they are in a topic by an offset — literally, the position of the last message they read. That offset is stored back in Kafka itself in a special topic called __consumer_offsets. This is elegant: Kafka uses itself to remember its own state.

Older versions of Kafka used a separate system called ZooKeeper to coordinate its brokers. Modern versions use KRaft — Kafka's own implementation of the Raft consensus algorithm — replacing ZooKeeper entirely and making the whole system simpler to operate.

60How LLMs are served

Serving a large language model at scale is different from serving almost any other kind of software, and understanding why is a small window into the current era of computing.

The essential fact is that generating each new token — each little chunk of text the model outputs — requires running the whole model, billions of parameters, once. Every response is many tokens, so the model runs many times per response. There is no shortcut; the arithmetic must be done.

Serving splits into two phases with different bottlenecks. The first phase, called prefill, is when the model reads your prompt. This is compute-bound — the GPU is doing arithmetic as fast as it can. The second phase, called decode, generates output tokens one at a time. Decode is memory-bandwidth-bound — the GPU spends most of its time reading model weights from memory. These two phases benefit from different optimizations, and modern serving systems handle them separately.

To keep GPUs busy, requests from many users are batched together. The GPU processes several conversations at once, which is much more efficient than serving them one by one. This is dynamic batching — the batch composition changes moment by moment as users come and go.

The KV cache is another critical piece. When the model reads a token, it computes something called attention state for that token. If we throw those states away, the next token's forward pass would have to recompute them. Instead, we save them. The KV cache holds all the attention states for the tokens generated so far. It uses a lot of GPU memory, but it saves an enormous amount of compute — without it, each new token would cost linearly more than the last.

When many users share the same long system prompt (say, an app's "you are a helpful assistant..." preamble), prompt-prefix caching avoids recomputing the prompt each time. Anthropic and OpenAI both expose this to developers as a first-class feature.

Speculative decoding is an increasingly common trick: a small, fast "draft" model proposes several tokens ahead; the big model then verifies them in parallel. When the draft is right (often), you get many tokens per big-model pass instead of one. When the draft is wrong, you lose a little and continue normally. Net effect: significant speedups on typical prompts.

Finally, quantization. Instead of storing model weights in full-precision 16- or 32-bit numbers, use 8-bit or even 4-bit numbers. The model gets slightly less accurate, but the memory footprint shrinks dramatically and inference becomes much cheaper. The trade-off is usually worth it for production.

61How a stock exchange works

A stock exchange is one of the few systems where microseconds matter and latency is a competitive weapon. Understanding one gives you a fresh appreciation for what "fast" really means in software.

At the core of every exchange is the order book — two sorted data structures per symbol. On one side, all the buy orders, sorted by highest price first (best offer to buy). On the other side, all the sell orders, sorted by lowest price first (best offer to sell). The gap between the best bid and the best ask is called the spread.

Matching is price-time priority. When a new order arrives, it's matched against the opposite side of the book in order of best price first, and among orders at the same price, the one that arrived earliest wins. So if you place a buy order for a hundred shares at ₹152, and the sell side has a fifty-share order at ₹152 (from Alice, placed earlier) and another at ₹152 (from Bob, placed later), your buy fills Alice first, then Bob.

Matching engines are almost always single-threaded per symbol. This sounds inefficient, but it's actually the point — single-threaded means deterministic. Every order that arrives in a given order produces the same result. That determinism is essential for regulators, for reconciling with clearing houses, and for auditing. Different symbols run on different engines, so overall throughput scales horizontally by running many engines in parallel.

The latency numbers are extreme: p99 in the tens of microseconds. Getting there requires abandoning most of what makes normal software easy. Trading firms use kernel bypass — network drivers that let packets skip the operating system's normal networking code (using libraries like DPDK). Some firms move matching logic onto FPGAs, custom hardware chips that run the logic in silicon rather than as CPU instructions. Firms rent rack space physically inside the exchange's data center to shave a few hundred microseconds. Microwave links between exchanges in different cities beat fiber optic cable because the signal travels faster through air over land than through glass.

After a trade matches (T+0), the trade is reported to regulators and to a clearing house — usually the DTCC in the US — which nets everyone's trades against each other and settles them a day or two later (T+1 or T+2). Custody of the actual shares changes hands at that settlement.

62How Google Docs collaborates in real time

Two people typing at the same place in the same document at the same time is the archetypal problem of collaborative software. Naively, one edit would clobber the other. Google Docs solves it with an algorithm called Operational Transformation, or OT.

The core idea: don't send the whole document back and forth. Send operations. An operation is a small object like "insert the character 'X' at position 12" or "delete 3 characters at position 20." When two people edit concurrently, both operations arrive at the server, but they may conflict — literally step on each other's positions. OT transforms one operation against the other so the final result matches what both authors intended.

An example: the document says "Hello." Alice inserts "X" at position 0. Simultaneously, Bob inserts "Y" at position 5 (after the "o"). Alice's operation and Bob's operation arrive at the server. The server picks an order — say Alice's first. Alice's insert doesn't need transformation; it goes in as-is, producing "XHello." But Bob's insert was written for a document without the "X" in front. If applied as-is, "Y" would land at position 5 of "XHello," which is between "l" and "o" — wrong. So OT transforms Bob's operation to insert at position 6 instead. The final document is "XHelloY," matching both authors' intent.

The server plays the role of the single source of truth for op order. Every operation is logged to persistent storage (Bigtable or Spanner) and the document is periodically snapshotted so replay doesn't have to start from scratch. Presence and cursor positions ride separately over WebSocket connections, updated with a short time-to-live so idle users are cleaned up.

Offline editing is handled by buffering ops locally. When you reconnect, your buffered ops are sent to the server, which transforms them against everything that happened while you were away and broadcasts the results to everyone.

Other collaborative apps — Figma, Linear, Apple Notes — solve the same problem with a different family of algorithms called CRDTs (Conflict-free Replicated Data Types). CRDTs are data structures that mathematically merge in any order, so you don't need a central server to coordinate. Both approaches work; OT tends to be a bit more efficient with server help, CRDTs enable fully decentralized collaboration.

63How Tinder scales matching

Tinder's core action is a swipe. About seventy-five million monthly users each swipe on the order of a hundred cards a day, adding up to something like seven and a half billion swipes per day. Underneath, three problems have to be solved fast: finding good candidates for you, recording swipes, and detecting matches the instant they happen.

The data model has three main tables. Swipes is the primary write log — every left or right swipe is a row with (user, target, direction, timestamp). Matches is a derived table populated whenever two users right-swipe each other. Profiles holds photos, age, distance preferences, and so on.

Candidate generation is where the recommendation problem lives. When you open Tinder and it hands you a batch of thirty cards, that batch was assembled by first bucketing candidates geographically (using geohashes — encoded strings where similar strings mean similar locations), then filtering by your age and other preferences, and finally re-ranking the top few hundred through a personalization model. The result is thirty cards personalized to you and served in a fraction of a second.

The match detection step is subtle. When you right-swipe someone, the app does two things: it writes your swipe row, and it checks whether the target already right-swiped you. That reverse check has to be O(1) — instant. Tinder stores, for each user, a Redis set called something like right_swipes_by:{target_id}, containing everyone who has right-swiped that target. When you right-swipe a target, the app checks whether you're in that target's Redis set. If yes, it's a match; both sides get a push notification instantly.

Recent activity matters. A user who was active in the last hour is boosted in the rankings for others. Tinder's business benefits when matches lead to conversations right away rather than a week later; the ranking is tuned toward that.

64How Nginx handles a hundred thousand connections

To appreciate Nginx you have to know what it replaced. Apache, the dominant web server of the 1990s and 2000s, used a thread-per-connection model. Ten thousand connected users meant ten thousand threads in memory. Each thread cost megabytes of stack space, and the operating system spent a lot of time context-switching between them. That model works for hundreds of connections and starts collapsing well before a hundred thousand.

Nginx took a different approach. Instead of one thread per connection, it uses one thread per CPU core — usually a handful — and multiplexes thousands of connections through each. The mechanism is an operating system feature called epoll on Linux (or kqueue on macOS and BSD). Epoll lets a single thread ask the kernel "wake me up when any of these thousands of sockets has something to do," and the kernel does. The thread wakes, handles the events, and goes back to waiting.

Because there's no thread per connection, memory per connection is tiny — around 2KB. A single Nginx server can hold well over a hundred thousand concurrent connections without breaking a sweat.

The Nginx process model has a master process that reads configuration and manages the workers, and a small number of worker processes that do the actual request handling. Each worker is single-threaded and runs its own epoll loop. Because workers don't share state, there's no contention between them.

Requests flow through a series of configurable phases: post_read, rewrite, access control, content generation, and logging. Each Nginx module — for compression, caching, SSL termination, whatever — hooks into whichever phase makes sense. The location blocks you write in Nginx configuration are really instructions about which modules run in which phases for which URL patterns.

When Nginx is used as a reverse proxy (its most common role), it keeps a pool of connections to your backend servers, buffers slow clients so the backend never sees them dawdling, and handles the mismatch between fast internal networks and slow user internet gracefully.

65Real-time leaderboards

Leaderboards are a beautiful little system-design problem because there's a data structure that fits the requirements almost too well: the Redis sorted set. A sorted set is a collection where every element has a numeric score and the collection is kept in sorted order automatically. All the operations you want on a leaderboard — add a score, get the top 100, look up a user's rank — are supported natively, and all run in O(log N) time.

Adding a score is one command: ZADD leaderboard 1500 user:42. Getting the top 100 is ZRANGE leaderboard 0 99 REV (the REV means descending, so highest scores first). Getting a user's rank is ZREVRANK leaderboard user:42. All fast, all built-in.

For millions of users on one leaderboard, a single Redis sorted set is fine. For tens of millions or hundreds of millions, you shard. Divide users by hash into buckets; each shard maintains its own sorted set; a leaderboard query hits each shard for its local top-K and merges the results. You accept that the merged top-K is eventually consistent — for a public leaderboard, this is perfectly fine.

Score events flow in through Kafka. A score processor drains them and updates the sorted sets. If updates come in faster than Redis can absorb, batch them — write to the sorted set every 500ms rather than on every event.

Time-decayed leaderboards (daily, weekly, all-time) are separate keys. A background job re-decays scores nightly or resets weekly boards on the appropriate day. Redis persistence — either AOF (append-only file) or periodic snapshots to durable storage — protects against data loss.

66How Apple AirTags find themselves

An AirTag is a very small device with a very short list of hardware: a coin-cell battery, a Bluetooth Low Energy radio, an accelerometer, and a UWB chip. No GPS. No cellular. No Wi-Fi. And yet if you lose one in a coffee shop in another country, you can open Find My on your phone and see where it is. The trick behind this is one of the more elegant privacy-preserving system designs of recent years.

The mechanism is a mesh of every iPhone on Earth. Your AirTag doesn't know where it is, but it periodically broadcasts a Bluetooth beacon with a rotating public key. Any nearby iPhone hears the beacon. That iPhone knows where it is (from its own GPS). It takes its own location and the current time, encrypts them with the AirTag's rotating public key, and uploads the encrypted blob to Apple's servers. Your iPhone, which holds the corresponding private key for your AirTag, later fetches the encrypted blob and decrypts it locally to reveal the AirTag's location.

The design does several clever things at once. Apple's servers store the encrypted blobs but cannot decrypt them — only the AirTag's owner can. The helper iPhone that reported the AirTag has no idea whose AirTag it was reporting because the public key rotates. Location tracking over time is defeated by the rotating keys — an observer watching beacons can't recognize that two beacons an hour apart came from the same tag.

When you get close to a missing item — within about thirty meters — a different chip takes over: UWB (ultra-wideband). UWB uses very precise time-of-flight measurements to determine direction with centimeter accuracy. This is the "arrow points to your keys" experience.

Anti-stalking is baked in. AirTags emit an audible chirp after being separated from their owner for a period of time. Both iPhones and Android phones (via a companion app) can detect unknown AirTags that seem to be traveling with a person and alert them.

67How Vitess scales MySQL to YouTube's numbers

It's often assumed that scaling a modern service requires abandoning SQL in favor of something more exotic. YouTube's core metadata database is a counterexample. It's plain old MySQL — but sharded across many machines by a project called Vitess, originally built inside YouTube and now open source under the Cloud Native Computing Foundation.

The Vitess architecture puts a smart proxy called VTGate in front of the database. Applications write ordinary SQL to VTGate as if it were a single database. VTGate parses the query, figures out which shard (or shards) the query needs to hit, forwards it there, aggregates results if necessary, and returns them.

Behind VTGate is a fleet of MySQL instances, each fronted by a small helper process called VTTablet. VTTablet enforces query rules, manages connections, and handles operational tasks like backups and failovers.

The features Vitess adds are exactly the ones you'd want if you were trying to scale MySQL. Transparent sharding means applications don't have to know how the data is distributed. Online resharding — splitting one shard into two — happens without downtime. Connection pooling turns thousands of application connections into a small number of MySQL connections, avoiding a common failure mode where the database dies from being asked to hold too many idle connections. Query rewriting catches expensive queries that would touch every shard and warns or rejects them. Automatic failover promotes a replica if a primary dies.

The lesson worth carrying away: scaling doesn't require throwing out the mature, well-understood tools. A smart proxy layer can hide a lot of complexity from the application while giving you all the scale you need. The same pattern applies to PostgreSQL through projects like Citus.
