PART 1 — SYSTEM DESIGN: FOUNDATIONS

1. What is System Design and how interviews are scored

What it is

System design is the art of taking a vague product requirement ("design Instagram") and producing a structured plan that explains:

  • What the system does (functional requirements)
  • How well it must do it (non-functional requirements — scale, latency, availability)
  • What pieces are involved (clients, servers, DBs, queues, caches)
  • How those pieces talk to each other (APIs, protocols)
  • How it survives failure and scale (replication, sharding, fallbacks)

What interviewers actually score (across Big Tech rubrics)

Signal What they look for
Requirements clarification Do you ask "what scale?", "read-heavy or write-heavy?", "consistency vs availability?" before drawing?
Structured approach Do you go: clarify → estimate → high-level → deep-dive → bottlenecks? Or do you randomly draw boxes?
Trade-off awareness Can you say "SQL gives strong consistency but sharding is painful; NoSQL scales easier but you lose joins"?
Depth in 1–2 areas Can you go deep on at least one component (the DB schema, the cache eviction, the queue consumer)?
Communication Are you driving the whiteboard or waiting to be asked? Are you thinking out loud?

The 5-step framework (memorize this)

1. CLARIFY      → Ask 4-6 questions. Pin down scale.
2. ESTIMATE     → Back-of-envelope: QPS, storage, bandwidth.
3. HIGH-LEVEL   → Draw the major boxes and arrows.
4. DEEP-DIVE    → Pick 1-2 components and go deep (schema, algo).
5. SCALE & FAIL → How does it scale 10x? What if X dies?

We'll come back to this in Section 55.

2. The Client–Server model

From absolute scratch

A client is anything that asks for data — your phone, your browser, an IoT device. A server is a program (usually running on a powerful computer in a datacenter) that listens for requests and sends back responses.

+--------+   request   +--------+
| Client | ----------> | Server |
|        | <---------- |        |
+--------+   response  +--------+

The client and server talk over a network (usually the Internet). They speak a shared protocol — most commonly HTTP.

Why this matters in interviews

Every system design question is, at its core, "how does the client get/send data to the server, and what happens server-side?" Even "design Uber" is just clients (riders, drivers) talking to servers, with extra state.

Types of clients you'll see in interviews

  • Web browser — HTML/JS app, talks HTTP
  • Mobile app (iOS / Android) — talks HTTP or WebSocket
  • Native desktop app — same as mobile
  • IoT device (kettle, scale, watch) — usually MQTT or HTTP
  • Another server (microservice-to-microservice) — gRPC, HTTP, or message queue

3. HTTP, REST, and API basics

HTTP from scratch

HTTP (HyperText Transfer Protocol) is a request/response protocol. The client sends a request and gets one response back. That's it.

A request looks like:

GET /users/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbGc...
Accept: application/json

A response looks like:

HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 87

{"id": 42, "name": "Lakshmi", "email": "lp@example.com"}

HTTP methods (verbs)

Method What it means Idempotent?
GET Read data Yes
POST Create something new No
PUT Replace existing (full update) Yes
PATCH Partial update No (technically)
DELETE Remove Yes

Idempotent = calling it twice has the same effect as calling it once. Important for retries! If a network blip causes your client to retry, you don't want to charge the user twice.

HTTP status codes you must know

Code Meaning
200 OK
201 Created (after POST)
204 No Content (after DELETE)
301 Moved permanently (redirect)
400 Bad Request (client sent garbage)
401 Unauthorized (not logged in)
403 Forbidden (logged in, but not allowed)
404 Not Found
409 Conflict (e.g. duplicate insert)
429 Too Many Requests (rate limited)
500 Internal Server Error
502 Bad Gateway
503 Service Unavailable
504 Gateway Timeout

REST (Representational State Transfer)

REST is a style of designing HTTP APIs around resources.

Bad (RPC-style):

POST /createUser
POST /getUserById
POST /deleteUserById

Good (REST):

POST   /users          → create a user
GET    /users/42       → fetch user 42
PATCH  /users/42       → update user 42
DELETE /users/42       → delete user 42
GET    /users/42/orders → list user 42's orders

REST best practices

  • Use nouns, not verbs, in URLs
  • Use HTTP methods to express the action
  • Use plural resource names (/users, not /user)
  • Version your API: /v1/users
  • Return appropriate status codes
  • Paginate list endpoints: /users?limit=20&cursor=abc

REST vs gRPC vs GraphQL (you'll be asked)

Aspect REST gRPC GraphQL
Transport HTTP/1.1 or 2 HTTP/2 HTTP/1.1
Payload JSON (usually) Protobuf (binary) JSON
Schema OpenAPI/Swagger .proto file GraphQL SDL
Streaming No (well, SSE) Yes (bi-di) Subscriptions
Best for Public APIs, mobile Internal microservices Flexible client queries
Over-fetching Yes No No (client picks fields)

WebSocket — when REST isn't enough

HTTP is request-response. The client always initiates. If the server wants to push data ("new message arrived!"), HTTP can't do it cleanly.

WebSocket keeps a persistent two-way pipe open. Used in:

  • Chat apps
  • Live dashboards
  • Multiplayer games
  • Real-time IoT telemetry

4. DNS and how a request actually reaches a server

When you type api.instagram.com into your phone, this happens:

1. App → OS:           "I want api.instagram.com"
2. OS → DNS resolver:  "Resolve api.instagram.com"
3. Resolver → Root DNS:  ".com is over there"
4. Resolver → .com DNS:  "instagram.com is over there"
5. Resolver → instagram DNS: "api → 52.84.123.45"
6. OS → app: "Connect to 52.84.123.45 on port 443"
7. App opens TCP connection → TLS handshake → HTTP request

Why interviewers care:

  • DNS is cached at multiple layers (OS, browser, resolver) → caching strategies generalize
  • GeoDNS can route a US user to a US server, India user to an India server — basis of CDN routing
  • DNS is also a single point of failure (multi-provider DNS exists for redundancy)

5. Load balancers (L4 vs L7)

The problem

One server can handle, say, 1000 requests/sec. You have 100,000 req/sec. So you need 100 servers. But the client only knows one address. How do requests get distributed?

Solution: Load Balancer

A box that sits between clients and servers, accepts incoming requests, and distributes them across a pool of backend servers.

                  +-------------+
   Client ----->  |  Load       |  ----> Server 1
   Client ----->  |  Balancer   |  ----> Server 2
   Client ----->  |             |  ----> Server 3
                  +-------------+        ...

Algorithms

Algorithm How it picks
Round Robin Rotate through servers 1, 2, 3, 1, 2, 3...
Least Connections Pick server with fewest active connections
Weighted Round Robin Bigger servers get more requests
IP Hash hash(client_ip) % N — same client always goes to same server (sticky sessions)
Least Response Time Pick the fastest server right now

L4 vs L7

  • Layer 4 (Transport): Routes based on IP + port. Fast, dumb. Doesn't understand HTTP. (e.g. AWS NLB, HAProxy in TCP mode)
  • Layer 7 (Application): Understands HTTP — can route based on path, headers, cookies. Slower, smarter. (e.g. AWS ALB, NGINX, HAProxy in HTTP mode)

Example L7 routing:

/api/users/*     → user-service backend pool
/api/payments/*  → payment-service backend pool
/api/*           → catch-all API pool
/static/*        → CDN

Health checks

Load balancers periodically ping each backend (e.g. GET /health every 5s). If 3 consecutive checks fail, that backend is removed from the pool. When it recovers, it's added back.

High availability of the load balancer itself

The LB is now a SPOF (single point of failure). Solutions:

  • Run 2+ LB instances in active-active or active-passive
  • Use a Virtual IP (VIP) that floats between them
  • Cloud LBs (AWS ALB/NLB) are managed and auto-redundant

6. API Gateways

What it is

An API Gateway is a smarter load balancer that sits at the edge of your system. It's the single front door for all clients.

It does:

  • Routing (/users → user service, /payments → payment service)
  • Authentication (validate JWT before forwarding)
  • Rate limiting (drop abusive clients)
  • Request/response transformation (e.g. v1 → v2 schema)
  • Logging & metrics
  • TLS termination (decrypt HTTPS so backends speak plain HTTP)
  • API key management
                    +---------------+
                    |               |
  Mobile  ----->    |               | ----> User Service
  Web     ----->    |  API Gateway  | ----> Payment Service
  IoT     ----->    |               | ----> Notification Service
                    |               |
                    +---------------+
                    [auth] [rate-limit] [logging]

Examples

  • AWS API Gateway
  • Kong
  • Apigee
  • NGINX (configured as a gateway)
  • Envoy
  • Spring Cloud Gateway

Gateway vs Load Balancer (common confusion)

  • LB: distributes traffic across identical backends.
  • Gateway: routes traffic to different backends and does cross-cutting concerns (auth, rate-limit). Often includes an LB.

7. Databases — SQL vs NoSQL

SQL (relational)

Data in tables (rows + columns). Relationships via foreign keys. Query with SQL.

users                      orders
+----+--------+           +----+---------+----------+
| id | name   |           | id | user_id | amount   |
+----+--------+           +----+---------+----------+
| 1  | Alice  |           | 11 | 1       | 250.00   |
| 2  | Bob    |           | 12 | 1       | 75.50    |
+----+--------+           | 13 | 2       | 1000.00  |
                          +----+---------+----------+

SELECT u.name, SUM(o.amount)
FROM users u JOIN orders o ON o.user_id = u.id
GROUP BY u.name;

Strengths: ACID transactions, joins, mature, well-understood, expressive queries. Weaknesses: Hard to scale horizontally (sharding is painful), schema changes are heavy.

Examples: PostgreSQL, MySQL, Microsoft SQL Server, Oracle.

NoSQL

An umbrella term for "not relational". Four main flavors:

a) Key-Value

Simplest. Like a giant hash map.

"user:42"  → {"name": "Lakshmi", "email": "..."}
"session:abc123" → {"user_id": 42, "expires": ...}

Examples: Redis, DynamoDB (KV mode), Memcached, etcd.

b) Document

Stores JSON/BSON documents. Schemaless.

{
  "_id": "user42",
  "name": "Lakshmi",
  "addresses": [{"city": "Bangalore"}, {"city": "Chennai"}]
}

Examples: MongoDB, CouchDB, Firestore.

c) Wide-Column

Rows can have wildly different columns. Optimized for very-large-scale writes.

Row key: "user42"
   col "name:first" → "Lakshmi"
   col "name:last"  → "P"
   col "login:2026-05-28" → "..."

Examples: Cassandra, HBase, ScyllaDB, BigTable.

d) Graph

Nodes + edges. Great for "friend of friend" queries. Examples: Neo4j, Amazon Neptune.

e) Time-Series (special case — important for IoT!)

Optimized for "many writes per second, queries by time range". Examples: InfluxDB, TimescaleDB, Prometheus.

Choosing in an interview

You need... Pick
Transactions, joins, structured data (users, orders, payments) SQL
Massive write throughput, simple lookups Wide-column (Cassandra)
Cache, session store, leaderboard Key-value (Redis)
Flexible schemas, JSON-heavy (e.g. product catalog) Document (MongoDB)
Social graph, recommendations Graph
Sensor / metric data Time-series

8. Indexes, transactions, ACID, BASE

Indexes

An index is like the back-of-the-book index — it lets the DB jump straight to the rows you want without scanning the whole table.

Without index:

SELECT * FROM users WHERE email = 'lp@example.com';
-- Full table scan. With 100M rows → slow.

With index on email:

CREATE INDEX idx_users_email ON users(email);
-- Now uses a B-tree lookup. O(log n).

Trade-off: indexes speed up reads but slow down writes (every insert/update must also update the index).

Common index types:

  • B-Tree: default, range queries (WHERE age > 25)
  • Hash: equality only (WHERE id = 42)
  • Composite: index on multiple columns (user_id, created_at) — order matters!

ACID (transactional guarantees)

Letter Meaning
Atomicity All-or-nothing. If part of a transaction fails, the whole thing rolls back.
Consistency DB moves from one valid state to another (constraints honored).
Isolation Concurrent transactions don't see each other's half-done work.
Durability Once committed, data survives crashes.

Isolation levels (you WILL be asked)

Level Allows
Read Uncommitted Dirty reads (almost never used)
Read Committed No dirty reads. (PostgreSQL default)
Repeatable Read Same query twice in a transaction = same result (MySQL default)
Serializable Strictest. Transactions appear to run one at a time.

BASE (the NoSQL counterpoint)

  • Basically Available
  • Soft state
  • Eventual consistency

"Don't expect ACID; expect that the system stays up and converges to a consistent state eventually."

9. Caching (and the famous cache strategies)

Why cache?

Reading from a DB might take 10–50ms. Reading from RAM (cache) takes <1ms. Real systems are 10–1000× faster with caching.

Where caches live

[ Browser cache ]
       |
[ CDN edge cache ]
       |
[ API Gateway cache ]
       |
[ Application cache (in-process) ]
       |
[ Distributed cache — Redis / Memcached ]
       |
[ Database internal cache ]
       |
[ Disk ]

Cache strategies (write side)

Write-through — write to cache AND DB synchronously.

client → write → cache → DB → ack

Pro: cache always fresh. Con: slow writes.

Write-behind (write-back) — write to cache, ack client, flush to DB later.

client → write → cache → ack
                   ↓ (async batch)
                  DB

Pro: fast writes. Con: data loss if cache dies before flush.

Write-around — write goes straight to DB; cache populated only on read miss.

client → write → DB
client → read  → cache miss → DB → populate cache → return

Pro: cache only holds hot data. Con: first read is slow.

Cache strategies (read side)

Cache-aside (lazy loading) — most common.

def get_user(id):
    user = cache.get(f"user:{id}")
    if user is None:
        user = db.query(id)
        cache.set(f"user:{id}", user, ttl=300)
    return user

Eviction policies (when cache is full)

  • LRU (Least Recently Used) — evict whoever hasn't been touched longest
  • LFU (Least Frequently Used) — evict the least-accessed
  • FIFO — first in, first out
  • TTL — eviction based on age

Cache problems (be ready to discuss)

Problem What Fix
Stale data Cache returns old value TTL, write-through, or invalidate on write
Thundering herd Cache expires, 1000 clients all hit DB Lock + single re-populate, or "stale-while-revalidate"
Cache penetration Lookups for keys that don't exist hammer DB Cache the "not-found" with short TTL, or bloom filter
Cache avalanche Many keys expire at the same time Add jitter to TTLs
Hot key One key (e.g. celebrity profile) overloads cache node Replicate across nodes; local in-process cache for top-N

10. Message queues and event-driven architecture

The problem

Service A wants to tell Service B "user just signed up — send them a welcome email." If A calls B directly:

  • A is slow if B is slow
  • A breaks if B is down
  • A can't easily fan out to C, D, E later

Solution: Queue

A message queue is a buffer in the middle.

Service A (Producer) → [ Queue ] → Service B (Consumer)
                                → Service C
                                → Service D

A drops a message and moves on. B reads when ready. If B is down, messages pile up safely. If B is slow, A is unaffected.

Queues vs Pub/Sub

  • Queue (point-to-point): each message consumed by exactly one consumer. Used for task distribution. (RabbitMQ, AWS SQS)
  • Pub/Sub (broadcast): each message delivered to every subscriber. Used for events. (Kafka, AWS SNS, Google Pub/Sub)

Kafka — the de-facto streaming platform

Kafka isn't really a queue — it's a distributed log. Messages are appended and retained for days/weeks. Multiple consumers each track their own offset.

Topic: "user-events"
Partition 0:  [m1][m2][m3][m4][m5] ...
Partition 1:  [m1][m2][m3]...
Partition 2:  [m1][m2][m3][m4][m5][m6]...

Consumer Group "email-service":  reads & advances offset
Consumer Group "analytics":      reads & advances independently

Why it's huge: ordering within a partition, replayability, horizontal scale, multiple consumers.

Delivery guarantees

Guarantee Meaning
At-most-once Might lose messages. Fast, no ack.
At-least-once Might duplicate. Common default. Consumer must be idempotent.
Exactly-once Hard. Kafka has a special mode; otherwise app-level dedup.

Patterns

  • Fan-out: one event, many consumers (email + push + analytics from a single sign-up event)
  • Worker pool: queue of tasks, many workers grab them in parallel
  • Event sourcing: store every state change as an event; rebuild state by replay
  • CQRS: separate write model (commands) from read model (queries)
  • Saga: long-running multi-step transactions across services using events + compensations

11. CDNs (Content Delivery Networks)

What

A network of servers around the world that cache static content close to users.

User in India  ─→ CDN edge in Mumbai (5ms)  ─→ (cache miss) → Origin in US (200ms)
User in US     ─→ CDN edge in Virginia (5ms)

What goes in a CDN

  • Images, videos, JS, CSS, fonts
  • Software downloads, OTA firmware (huge for IoT!)
  • Sometimes API responses with short TTL

How invalidation works

  • TTL-based (expire after N hours)
  • Purge by URL when you update the asset
  • Cache busting with hashed filenames: app.a3f8b9.js

Examples

Cloudflare, AWS CloudFront, Akamai, Fastly.

12. Object storage (S3-style)

What

A service that stores blobs (files) addressed by a key, accessible over HTTP. Cheap, massively scalable, durable.

PUT  /bucket-name/path/to/file.jpg
GET  /bucket-name/path/to/file.jpg
DELETE /bucket-name/path/to/file.jpg

Why not just store in DB?

  • DBs are expensive per GB; object storage is cents per GB
  • DBs choke on multi-MB blobs
  • Object storage is designed for "write once, read many"

Use cases

  • User uploads (photos, videos, documents)
  • Backups
  • Static website hosting
  • Data lake / analytics
  • OTA firmware images

Concepts

  • Buckets: top-level containers
  • Objects: the files themselves
  • Versioning: keep old versions on overwrite
  • Lifecycle policies: auto-move old data to colder/cheaper storage (S3 → Glacier)
  • Presigned URLs: time-limited URLs to upload/download without exposing credentials

Examples: AWS S3, GCS, Azure Blob, MinIO (self-hosted).

13. Rate limiting (algorithms + implementations)

Why

  • Prevent abuse / DoS
  • Fair usage across tenants
  • Protect downstream services from being overwhelmed

Algorithms

1. Fixed Window "Max 100 requests per minute starting at :00, :01, :02..."

counter[user][minute_bucket]++
if counter > 100: reject

Pro: simple. Con: spike at boundary (100 at 0:59 + 100 at 1:00 = 200 in 2s).

2. Sliding Window Log Keep a timestamp for every request in the last 60s. Count them. Pro: accurate. Con: memory-heavy.

3. Sliding Window Counter Weighted blend of "current minute counter" + "previous minute counter". Pro: low memory, smooth. Con: approximate.

4. Token Bucket Bucket holds N tokens. Refills at rate R per second. Each request takes 1 token. If empty → reject.

tokens = min(capacity, tokens + (now - last_refill) * rate)
if tokens >= 1: tokens-- ; allow
else: reject

Pro: allows bursts up to bucket size, then smooths out. Most popular.

5. Leaky Bucket Queue of fixed size. Drains at constant rate. Overflow → drop. Pro: smooth output. Con: no burst.

Distributed rate limiting

Counter must be shared across all API gateway instances. Common store: Redis with INCR + EXPIRE.

INCR rate:user:42:202605281530
EXPIRE rate:user:42:202605281530 60

If the value > limit → reject with HTTP 429.

We design a full distributed rate limiter in Section 28.

14. Authentication & Authorization

Terms

  • Authentication (AuthN): who are you?
  • Authorization (AuthZ): what can you do?

Session-based auth

1. Client POSTs username/password
2. Server creates session_id, stores it server-side (Redis/DB)
3. Server sets cookie: Set-Cookie: session=abc123
4. Every later request carries the cookie
5. Server looks up session_id → user_id → grants access

Pro: easy to revoke (delete the session row). Con: needs server state — annoying across many servers.

Token-based auth (JWT)

A JWT is a signed string the client carries:

header.payload.signature

eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiI0MiIsImV4cCI6MTcwMDB9.SflKxw...

Payload (decoded):

{"sub": "42", "exp": 1700000000, "roles": ["user"]}

The server signs it with a secret. On every request, it just verifies the signature — no DB lookup. Stateless.

Pro: scales without server-side session storage. Con: hard to revoke before expiry (use short TTL + refresh tokens).

OAuth 2.0 (you'll be asked)

"Log in with Google" — your app never sees the user's Google password.

1. User clicks "Login with Google"
2. App redirects to Google with: client_id, scope, redirect_uri
3. User logs into Google, approves access
4. Google redirects back to your app with a code
5. App's backend exchanges code + client_secret → access token
6. App calls Google APIs with token to get profile

Authorization patterns

  • RBAC (Role-Based): user has roles → roles have permissions
  • ABAC (Attribute-Based): rules over attributes ("user can edit if user.id == doc.owner_id")
  • ACL: explicit "user X has perm Y on resource Z" rows

15. Scaling — vertical, horizontal, replication, sharding

Vertical scaling (scale up)

Bigger machine. 4-core → 32-core. 16GB → 256GB. Easy. Limited — there's a max CPU/RAM you can buy, and it's expensive.

Horizontal scaling (scale out)

More machines. 1 server → 100 servers behind a load balancer. Practically unlimited, but introduces complexity (state synchronization, sharding).

Stateless vs Stateful servers

  • Stateless: any request can go to any server. Easy to scale. Push state to DB/cache.
  • Stateful: certain requests must go to a specific server (e.g. WebSocket connection). Harder to scale.

Database scaling

Replication (read scaling) 1 primary handles writes. N replicas handle reads. Replicas lag behind by ms-to-seconds.

Writes →  Primary
            ↓ async
            ├→ Replica 1 → Reads
            ├→ Replica 2 → Reads
            └→ Replica 3 → Reads

Strong-consistency reads must hit primary. Eventual-consistency reads can hit any replica.

Sharding (write + storage scaling) Split data across N DBs by some key.

shard_id = hash(user_id) % N
  • Range sharding: A–F → shard1, G–M → shard2... (hot ranges = bad)
  • Hash sharding: hash(key) % N (uniform, but resharding is painful)
  • Consistent hashing: hash onto a ring; adding a shard only moves 1/N of data (used by Cassandra, DynamoDB)

Cross-shard queries are PAINFUL. Design your shard key around your hottest query patterns.

Consistent hashing — visualized

                hash ring (0 → 2^32)
                       0
                    /     \
              Node A       Node B
                |             |
            (key1, key2)   (key3, key4)
                |             |
              Node D ─────── Node C
                |             |
            (key5)         (key6, key7)

Each key is placed at its hash position, then assigned to the next node clockwise. Adding/removing a node only reshuffles the keys in that node's slice.

16. Consistency, Availability, CAP, PACELC

CAP theorem (in one sentence)

When a network partition happens, a distributed system must choose between Consistency and Availability — you can't have both.

  • Consistency: every read returns the latest write.
  • Availability: every request gets a response (maybe stale).
  • Partition tolerance: works despite network splits.

P isn't optional in real systems (the network will fail), so you choose CP or AP.

DynamoDB AP (tunable)

PACELC (the smarter version)

If there is a Partition, choose A or C. Else (normal operation), choose between Latency and Consistency.

Even with no partition, replication has a cost: do you wait for all replicas to ack (slow, consistent) or return immediately (fast, eventual)?

Consistency models (spectrum)

  • Eventual: replicas converge eventually
  • Read-your-writes: you see your own updates immediately
  • Linearizable: as if there's one global ordered log of operations

18. URL Shortener (TinyURL)

Requirements

Functional

  • Shorten a long URL → return short URL (e.g. tinyurl.com/aB3xZ9)

Non-functional

  • 10:1 read:write ratio → 12,000 reads/sec
  • Read latency p99 < 100ms
  • 99.9% availability
  • URLs stored ~5 years

Estimates

  • Writes/day: 100M
  • 5 years: ~180B URLs
  • Per URL: ~500 bytes (long URL + meta) → ~90 TB total
  • Short code: 7 chars in base62 = 62^7 ≈ 3.5 trillion combinations (plenty)

High-level

   +---------+        +-----------+         +----------+
   | Client  |  --->  |  API/LB   |  --->   | App tier |
   +---------+        +-----------+         +----------+
                                              |    |

                                  |                       |
                              +-------+              +----------+
                              | Redis |              | Database |
                              | cache |              | (shards) |
                              +-------+              +----------+

Generating short codes — three approaches

A. Hash the long URL (e.g. MD5, take first 7 chars)

  • Pro: deterministic (same URL → same code)
  • Con: collisions; handle with retry or random suffix

B. Counter + base62 encode

  • Maintain a global counter. For each new URL: id++ → base62(id) → "aB3xZ9"
  • Pro: no collisions
  • Con: counter is a bottleneck. Solve with: per-shard counters, or a range pre-allocator (each app server pre-allocates a batch of 1000 IDs)

C. Random + collision check

  • Generate random 7-char string, check DB, retry on collision
  • Pro: simple
  • Con: extra DB lookup per write

Recommended: B (counter with batched ID allocation) — simple, no collisions, scalable.

Schema

urls (
  short_code   VARCHAR(10) PRIMARY KEY,
  long_url     TEXT NOT NULL,
  user_id      BIGINT,
  created_at   TIMESTAMP,
  expires_at   TIMESTAMP NULL,
  click_count  BIGINT DEFAULT 0
)
INDEX idx_user (user_id)

     hit → 301 redirect to long_url
     miss → db.get("aB3xZ9")
            → cache.set, return 301

Write path

POST /shorten {url: "..."}
  → validate URL
  → allocate next ID (from local batch)
  → base62(id) = short_code
  → db.insert(short_code, url, ...)
  → cache.set(short_code, url)
  → return short_code

Scale & failures

  • Cache layer: Redis cluster, LRU eviction. Hot URLs (a viral link) stay hot.
  • DB: sharded by short_code. Cassandra works well here (high write throughput, key-value access pattern).
  • CDN: put the redirect endpoint behind CloudFront with short TTL.
  • Analytics: don't write to DB on every click — push to Kafka, aggregate hourly into a separate analytics DB.

Follow-ups interviewers ask

  • "What if the same URL is shortened twice?" → check hash, return existing
  • "How do you handle custom aliases?" → user provides code; check uniqueness; reserve namespace
  • "Expiry?" → background sweeper job, or TTL on cache/DB
  • "How to prevent abuse?" → rate-limit by IP/user, malware URL detection

Location indexing — the heart of Uber

You need "find 10 drivers within 2 km of point P" fast.

Options:

  • Geohash: encode lat/lng as a string. Prefix match = nearby.
  • Quadtree: recursive 2D partition
  • S2 (Google): maps Earth to a 1D Hilbert curve over cells

1. Rider requests ride at point P
2. Query "drivers within radius R of P, available, vehicle type matches"

24. File Upload System (Dropbox / Google Drive)

Functional

  • Upload / download files
  • Sync across devices
  • Share with users
  • Versioning, deleted-file recovery

Key design ideas

Chunking Split each file into 4MB chunks. Upload chunks in parallel. Resumable on failure.

Deduplication Hash each chunk (SHA-256). If hash already exists in storage → reference it, don't re-upload. Massive storage savings.

Schema (simplified)

files(file_id, owner_id, name, size, root_chunk_list_id, version)
chunks(chunk_hash PRIMARY KEY, s3_url, ref_count)
file_versions(file_id, version, chunk_list)

Upload flow

1. Client splits file into chunks, computes hashes locally
2. Client POSTs metadata + hash list
3. Server returns: "these 5 chunks I already have, please upload these 3"
4. Client uploads missing chunks to S3 via presigned URLs
5. Server commits file version

Sync (multi-device)

Each device has a local DB of (file_id, version). Periodically polls (or listens on WS) for changes. Server returns delta.

Conflict resolution: last-write-wins for simple cases; create file (conflicted copy) for collisions.

Scale

  • Metadata: sharded SQL by user_id (you almost never query across users)

Requirements

  • Process charges
  • Refunds
  • Idempotency (critical!)
  • Strong consistency
  • Audit log

Non-negotiables

  • Idempotency: client sends Idempotency-Key. If duplicate → return the same response. Never charge twice.
  • Atomicity: balance debit + ledger insert in one DB transaction.
  • Audit log: every state change is an append-only event in a separate store.

Client → POST /charge (Idempotency-Key: abc, amount, payment_method)
  → API checks key in dedup store
      hit → return prior response
  → Start DB transaction
       insert payment(pending)

            external API — careful about partial state
       on gateway success: update payment(completed)
       insert ledger entry

Every transaction = two entries.

28. Distributed Rate Limiter

Goal

Architecture

Gateway 1                Gateway 2              Gateway 3
    |                        |                      |
    +-------------+----------+----------------------+
                  |
                  v
              Redis cluster   ← counters

Token bucket in Redis (Lua script for atomicity)

-- KEYS[1]: "rl:user:42"
-- ARGV[1]: now (ms)
-- ARGV[2]: refill_rate (tokens/ms)
-- ARGV[3]: capacity
-- ARGV[4]: cost (usually 1)

local data = redis.call('HMGET', KEYS[1], 'tokens', 'last')
local tokens = tonumber(data[1]) or tonumber(ARGV[3])
local last = tonumber(data[2]) or tonumber(ARGV[1])
local now = tonumber(ARGV[1])

tokens = math.min(tonumber(ARGV[3]),
                  tokens + (now - last) * tonumber(ARGV[2]))

if tokens >= tonumber(ARGV[4]) then
  tokens = tokens - tonumber(ARGV[4])
  redis.call('HMSET', KEYS[1], 'tokens', tokens, 'last', now)
  redis.call('EXPIRE', KEYS[1], 120)
  return 1   -- allow
else
  return 0   -- deny
end

Optimizations

  • Local cache + sync: each gateway tracks its own bucket and reconciles with Redis every 100ms. Reduces Redis QPS at the cost of slight over-allowance.
  • Sharded counters: shard by user_id across Redis cluster to spread load.
  • Tiered limits: per-user, per-IP, per-tenant, global.

Failure modes

  • Redis down → fail open (allow all) or fail closed (deny all)? In practice: fail open with circuit breaker + local backup limit.

31. Sync System Design

Problem

Multiple devices (phone, watch, tablet) and a server. User changes data on phone offline. How does it land on watch and server, and how do conflicts resolve?

Key concepts

Vector clocks / Lamport timestamps: order events without a global clock.

Last-write-wins (LWW): simplest. timestamp_a > timestamp_b → a wins. Works for many use cases (notes, settings).

CRDTs (Conflict-free Replicated Data Types): data types that mathematically guarantee merge convergence (G-Counter, OR-Set, RGA for text editing). Used by Figma, Apple Notes, Linear, Riak.

Operational Transformation (OT): used by Google Docs. Each edit is an operation; concurrent ops are transformed against each other.

Sync protocol (typical)

Device                       Server
  | GET /sync?since=<token>     |
  | -----------------------→    |
  |    [diff of changes,        |
  |     new sync_token]         |
  | ←-----------------------    |
  |                             |
  | apply remote changes        |
  | resolve conflicts (LWW/CRDT)|
  |                             |
  | POST /sync                  |
  | { local_ops: [...] }        |
  | -----------------------→    |
  |     ack + new_sync_token    |
  | ←-----------------------    |

Storage shape

Each item carries: (id, value, modified_at, modified_by_device, deleted_flag). Soft-delete (deleted_flag) — never hard-delete until tombstone TTL.

Real-time push

Plain polling wastes battery. Use silent push (APNs/FCM) to wake the app, which then runs a /sync call.

                       +------------------+

Outbox pattern

  • Mutation: write to (notes, status='dirty') + insert into outbox.
  • Sync engine reads outbox FIFO, posts to server, on success marks clean.
  • On failure: retry with exponential backoff. After N attempts: surface to user.

Conflict

  • Server may have changed the same row → LWW or surface conflict UI.
  • Watch out for stale device clocks; use server-issued timestamps where possible.

Battery-friendly sync triggers

  • App foreground
  • Significant location change
  • Silent push from server
  • Background fetch (iOS: low-frequency budget)
  • Network changed from cellular → Wi-Fi (good time to sync big stuff)

(Section 20 covered the general system. Here, the mobile-specific deep dive.)

iOS — APNs

Throughput

  • APNs: ~9,000 messages/sec per HTTP/2 connection; open many in parallel

34. BLE / Wi-Fi Device Setup System

The problem (very common at IoT companies)

User unboxes a smart kettle. It has no display, no keyboard, no Wi-Fi credentials. How does it join the home Wi-Fi?

Common flows

A. BLE provisioning (most common today)

1. Device boots into "setup mode" — advertises BLE GATT service
2. User opens app, app scans BLE, finds device
3. App connects over BLE (encrypted with ECDH key exchange)
4. App reads list of Wi-Fi SSIDs the device sees
5. User picks SSID, enters password in app
6. App sends Wi-Fi credentials over BLE
7. Device disconnects BLE, connects to Wi-Fi
8. Device calls cloud /devices/claim → returns a device cert
9. App polls /devices/{id}/status → "connected"

B. Soft-AP provisioning (older)

  • Device exposes its own Wi-Fi AP (Kettle-XXYY)
  • Phone joins that AP → loses Internet
  • App sends creds over HTTP to device
  • Device joins user's Wi-Fi

C. Wi-Fi Easy Connect (DPP) (newer)

  • QR-code-based handshake

Sketch — BLE setup

+---------+   BLE   +---------+   Wi-Fi   +---------+    HTTPS/MQTT   +-------+
|  App    | <-----> | Device  | <-------> | Router  | <---------------> Cloud|
+---------+         +---------+           +---------+                  +-----+
                       ^
                       │  GATT services:
                       │   - Setup     (WiFi SSID, password)
                       │   - Status    (state, error code)
                       │   - Identity  (serial, fw version)

Security

  • No plaintext credentials over BLE. Use static BLE key (from QR code on device) or ECDH key exchange.
  • Device certificates: pre-provisioned at factory, or just-in-time (JITP) via AWS IoT.
  • Replay protection: nonces in BLE handshake.
  • Claim flow: app proves device ownership to cloud (e.g. by reading a one-time code from the device over BLE and posting to cloud).

Onboarding state machine

UNCLAIMED → SETUP_MODE → WIFI_CONNECTING → CLOUD_CONNECTING → CLAIMED → ONLINE
                            ↓                ↓                          ↓
                          ERROR            ERROR                     OFFLINE

App polls cloud to drive UI through these states.

Failure modes

  • Wrong Wi-Fi password → device reports specific error code over BLE before disconnecting
  • Wi-Fi joins but cloud unreachable → retry, surface "check internet" to user
  • Phone loses BLE mid-setup → device times out and reverts to setup mode

Architecture

Devices  ──MQTT──→  [Edge MQTT Broker]
                          │
                          v
                  ┌──────────────────┐
                  │   IoT Gateway    │  (auth, deduplication, rate limit)
                  └──────────────────┘
                          │
                          v

                          │

Why MQTT?

  • Designed for unreliable networks (mobile, IoT)
  • Persistent session (broker buffers while device offline)
  • QoS levels: 0 (fire-and-forget), 1 (at-least-once), 2 (exactly-once)
  • Tiny header (2 bytes minimum)

Schema (Influx-style line protocol)

                  └── tags (indexed) ──┘            └ field ┘  └ timestamp ──┘

Stream processing — typical jobs

  • Per-device 1-minute average → write back to Influx
  • Anomaly detection (Z-score over rolling window) → emit alert event
  • Device offline detection (no heartbeat in 5 min) → alert
  • Cumulative usage metric (kWh today) → user dashboard

Backpressure

  • If downstream slow, MQTT broker buffers, Kafka topic absorbs spikes (retention 7d)
  • Consumers scale horizontally (one per Kafka partition)

Failure / loss tolerance

  • Devices buffer locally if disconnected; flush on reconnect (small ring buffer in flash)
  • Kafka replication factor 3
  • TS DB replicated cluster

Cost knobs

  • Sample rate (1Hz vs 0.1Hz)
  • Aggregation tier (raw 30d → 1-min avg forever)
  • Compression (Parquet + zstd for cold)

37. Time-Series Data Storage

What makes TS DBs special

  • Append-only writes (no updates)
  • Queries are almost always "range over time + group by tag"
  • Heavy compression possible because adjacent timestamps and values are correlated

Internals (InfluxDB / TimescaleDB-style)

  • Hypertables: virtual table partitioned by time (e.g. 1 day chunks)
  • Columnar storage: timestamps, values stored in columns → great compression
  • Tag index: inverted index tag_key=value → list of series
  • Retention policies: drop chunks older than N days

Compression tricks

  • Delta encoding: store t1, t2-t1, t3-t2, ...
  • Delta-of-delta: works because intervals are often constant (1s, 1s, 1s)
  • Gorilla compression (Facebook): 12 bytes/sample → ~1.4 bytes/sample
  • Run-length for tag dictionaries

When NOT to use TS DB

  • Heavy update / random access
  • High-cardinality dimensions (user_id as a tag → explodes)

38. OTA Firmware Update System

Requirements

  • Push new firmware to fleet of N devices
  • Staged rollout (1% → 10% → 100%)
  • Rollback on failure
  • No bricks (atomic update)
  • Signed images (security)

Architecture

Build pipeline → S3 (signed .bin) → Manifest service ──┐
                                                       v
                                                 +-----------+
   Device ----- check-for-update -------------→ |  Cloud    |
   Device ←---- response (or none) ---------- |  Manager   |
   Device ----- download chunks ─→ CDN          +-----------+
   Device ----- report progress/status ────→

Manifest example

{
  "fw_version": "2.4.0",
  "url": "https://cdn/.../kettle-2.4.0.bin",
  "sha256": "...",
  "signature": "...",            // signed with vendor private key
  "min_hw_revision": "B",
  "rollout": {"percent": 10}
}

Device-side update flow

1. Device checks /firmware/check?fw=2.3.1&hw=B&serial=...
2. Server applies rollout rules → returns 2.4.0 manifest or no-update
3. Device verifies signature on manifest
4. Device downloads .bin in chunks (resume on network drop)
5. Verifies SHA-256 + image signature
6. Writes to inactive partition (A/B partitions)
7. Sets boot flag → reboot
8. Bootloader boots new partition
9. New firmware runs self-check; if OK → marks as confirmed
10. If self-check fails or device fails to "check in" within N reboots → bootloader rolls back to previous partition

Staged rollout

  • Bucket each device into 0–99 by hash(serial)
  • Server says "give me 2.4.0 only if bucket < 10" (10%)
  • Watch crash/return-rate metrics → expand bucket

Why A/B partitions

  • Atomic switch (just change boot pointer)
  • Rollback = boot the other partition
  • Critical for safety: never overwrite the running firmware

Bandwidth optimization

  • Delta updates (bsdiff / xdelta): ship just the diff vs prior version
  • CDN required (firmware is MBs, fleet is millions)

Security

  • All images signed by vendor key burned into the chip's secure boot
  • TLS for download
  • Manifest signed too

This is essentially what your smart-kettle firmware project will need — call it out in interviews.

40. OOP Fundamentals From Scratch

What is an object?

A bundle of state (fields) and behavior (methods) that represents one thing.

class Car:
    def __init__(self, make, model):
        self.make = make         # state
        self.model = model
        self.speed = 0

    def accelerate(self):        # behavior
        self.speed += 10

Four pillars

1. Encapsulation — hide internal state behind methods. Callers can't poke fields directly.

class Account:
    def __init__(self):
        self.__balance = 0       # private
    def deposit(self, amt):
        if amt > 0:
            self.__balance += amt
    def get_balance(self):
        return self.__balance

2. Abstraction — expose what an object does, hide how. A List exposes add, remove, size. You don't know if it's an array or a linked list under the hood.

3. Inheritance — a class can extend another.

class Vehicle: ...
class Car(Vehicle): ...
class ElectricCar(Car): ...

Use sparingly. Prefer composition (object holds other objects) over deep inheritance.

4. Polymorphism — same interface, different behavior.

class Shape: 
    def area(self): pass
class Circle(Shape):
    def area(self): return 3.14 * self.r ** 2
class Square(Shape):
    def area(self): return self.s ** 2

for shape in [Circle(5), Square(3)]:
    print(shape.area())   # works for both

Class diagrams (UML) — minimum to know

+-------------------+
|     Account       |     ← class name
+-------------------+
| -balance: float   |     ← - private, + public, # protected
+-------------------+
| +deposit(amt)     |
| +withdraw(amt)    |
| +get_balance()    |
+-------------------+

Account  ◇──── Customer       composition (filled diamond = strong owner)
Account  ◆──── Customer       aggregation (open diamond = weak)
Manager ──▷ Employee          inheritance
Order  ----▶ Product          association ("uses")

41. SOLID Principles

Five rules that make OO designs reusable and maintainable. Asked verbatim in interviews.

S — Single Responsibility

A class should have one reason to change.

Bad: User knows how to validate itself AND persist to DB AND send email. Good: User (data) + UserValidator + UserRepository + WelcomeEmailer.

O — Open/Closed

Open for extension, closed for modification.

Bad: adding a new payment type requires editing the PaymentProcessor switch statement. Good: PaymentProcessor takes a PaymentMethod interface; new methods implement the interface without touching the processor.

L — Liskov Substitution

Subtypes must be usable wherever their parent type is expected.

Classic violation: Square extends Rectangle. Setting width changes height, breaking callers that assumed independent dimensions. → Don't model "is-a" when behavior differs.

I — Interface Segregation

Don't force clients to depend on methods they don't use.

Bad: IPrinter with print(), scan(), fax(). A scanner-only device must implement print() as a stub. Good: separate IPrint, IScan, IFax.

D — Dependency Inversion

Depend on abstractions, not on concretions.

Bad: OrderService directly new MySqlDb(). Good: OrderService(repo: OrderRepository). The concrete MySqlOrderRepository is injected.

42. Core Design Patterns

The Gang-of-Four patterns split into three buckets. Memorize the top ~12.

Creational

Singleton — exactly one instance of a class.

class Logger:
    _instance = None
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance

Use sparingly. Hard to test. Prefer DI.

Factory — encapsulate object creation.

def make_shape(kind):
    if kind == 'circle': return Circle()
    elif kind == 'square': return Square()
    raise ValueError(kind)

Builder — step-by-step construction of a complex object.

pizza = (PizzaBuilder()
         .size('large')
         .topping('cheese')
         .topping('mushroom')
         .build())

Prototype — clone an existing instance.

Structural

Adapter — make an incompatible interface fit.

ThirdPartyAPI  ←  AdapterClass  ←  YourCode (uses YourInterface)

Decorator — wrap an object to add behavior without subclassing.

class Logger:
    def log(self, msg): print(msg)

class TimestampedLogger:
    def __init__(self, inner): self.inner = inner
    def log(self, msg): self.inner.log(f"{now()} {msg}")

Facade — single simple front door over a complex subsystem.

Proxy — stand-in object (lazy load, access control, remote).

Composite — tree structures (folder contains folders + files; all implement size()).

Behavioral

Observer — subjects notify subscribers on state change.

class Subject:
    def __init__(self): self.observers = []
    def subscribe(self, o): self.observers.append(o)
    def notify(self, event):
        for o in self.observers: o.on_event(event)

Strategy — interchangeable algorithms via an interface.

class SortStrategy: 
    def sort(self, data): pass
class QuickSort(SortStrategy): ...
class MergeSort(SortStrategy): ...

class Sorter:
    def __init__(self, strategy): self.strategy = strategy
    def run(self, data): return self.strategy.sort(data)

State — object behavior changes with internal state (often replaces big switch statements).

OrderState ─── PendingState, PaidState, ShippedState, DeliveredState
order.handle("pay") → state.handle(order) → transitions to PaidState

Command — wrap an action as an object (enables undo, queueing).

Iterator — uniform way to traverse a collection.

Template Method — base class defines algorithm skeleton; subclasses fill steps.

Chain of Responsibility — request travels along a chain until one handler handles it. (Used in middleware, exception handling.)

43. Dependency Injection

DI is just: "don't new your dependencies inside your class; have them passed in."

# Bad
class OrderService:
    def __init__(self):
        self.db = MySqlDB()           # hardwired
        self.emailer = SmtpEmailer()  # hardwired
    def place_order(self, ...):
        self.db.save(...)
        self.emailer.send(...)

# Good
class OrderService:
    def __init__(self, db: OrderRepo, emailer: Emailer):
        self.db = db
        self.emailer = emailer

In tests:

order_service = OrderService(InMemoryOrderRepo(), FakeEmailer())

Frameworks (Spring, Dagger, Hilt, Swinject, NestJS) automate the wiring.

44. MVC vs MVVM vs Clean Architecture

MVC (Model-View-Controller)

View  ─── user action ──→  Controller ── updates ──→  Model
View  ←── render ────────  Controller ←── notify ───  Model

MVVM (Model-View-ViewModel) — iOS/SwiftUI default

View  ─── user action ──→  ViewModel  ── reads ──→  Model
View  ←── binding ──────   ViewModel  ←── observes  Model

ViewModel exposes observable state; View binds to it. Easier to unit test (no UI).

Clean Architecture

Layers, with dependencies pointing inward only:

Outer:   UI / Frameworks
            ↓ depends on
         Interface Adapters (presenters, controllers, gateways)
            ↓
         Use Cases / Interactors (application business rules)
            ↓
         Entities (enterprise business rules)   ← innermost, knows nothing of outer

Use cases depend on interfaces the outer layers implement. Maximum testability.

45. Thread-Safe Design

Concepts

  • Race condition: two threads access shared state and result depends on timing.
  • Critical section: code that touches shared state.
  • Mutex / Lock: only one thread in critical section at a time.
  • Semaphore: up to N threads at once.
  • Atomic: single-instruction operations (incr, compare-and-swap).
  • Read-Write Lock: many readers, one writer.
  • Lock-free: use atomics + CAS loops to avoid locks (advanced).

Common patterns

Producer-Consumer (bounded queue)

producer ──→ [queue]  ──→ consumer

put() blocks when full; get() blocks when empty.

Read-write split Readers don't block readers; writers exclude both.

Immutability Immutable objects are free to share across threads. (final in Java, let in Swift, const in JS).

Pitfalls

  • Deadlock: thread A holds X, waits for Y; thread B holds Y, waits for X. Fix: always acquire locks in the same global order.
  • Livelock: threads keep yielding to each other, no progress.
  • Starvation: low-priority thread never runs.
  • False sharing: two atomics on the same cache line → contention.

In tests

Use deterministic schedulers or property-based tests. Never use "sleep + hope".

     Kafka for fan-out")
   - Check in: "Anything you'd like me to focus on?"

0:20 - 0:35    DEEP-DIVE
   - Pick 1-2 components. GO DEEP.
   - DB schema with columns
   - Indexes, sharding key, replication factor
   - Concrete algorithm (hash function, eviction policy)
   - Specific failure scenario + recovery

0:35 - 0:42    SCALE & FAILURE MODES
   - "How does this handle 10x the load?"
   - "What happens if the DB primary dies?"
   - "What about a region outage?"
   - Mention monitoring/alerting briefly.

0:42 - 0:45    WRAP & TRADE-OFFS
   - Recap key trade-offs you made
   - Mention 1-2 things you'd improve with more time
   - Invite questions

58. Mistakes That Get Candidates Rejected

  1. Jumping into design before clarifying scope. Always clarify first.
  2. Drawing 15 boxes, going deep on zero. Pick 1–2 and go deep.
  3. No trade-offs. Every choice has a cost; name it.
  4. Hand-waving the hard part. "And then we just shard it." How?
  5. Ignoring failure modes. What if the DB dies? The cache dies? The region dies?
  6. Wrong DB for the job. "I'd use Mongo for ACID transactions" → no.
  7. Capacity math missing. Estimates make you concrete.
  8. Silence. Interviewer can't score your thinking if they can't hear it.
  9. Defensiveness on follow-ups. Treat every "what if" as a gift, not an attack.
  10. Memorized solution + wrong question. They'll twist the problem; adapt.

(Use these as flash cards.)

Load Balancer — distributes traffic. L4 (TCP) fast/dumb, L7 (HTTP) slow/smart. Algorithms: RR, LC, weighted, IP hash. Health-checks backends. Itself needs HA.

API Gateway — front door. Routes + auth + rate-limit + transform + log. AWS API GW, Kong, Apigee.

SQL — ACID, joins, mature. Hard to scale horizontally. Use for: transactions, structured data.

NoSQL — KV (Redis), Document (Mongo), Wide-column (Cassandra), Graph (Neo4j), TS (Influx).

Cache — RAM-fast. Strategies: write-through, write-back, write-around, cache-aside. Eviction: LRU/LFU/TTL. Pitfalls: thundering herd, penetration, avalanche, hot key.

Queue — buffer. Producer/consumer. Kafka = distributed log. SQS/RabbitMQ = traditional. Delivery: at-most/at-least/exactly-once. Make consumers idempotent.

CDN — geo-distributed static cache. CloudFront, Cloudflare, Akamai.

Object storage — blobs over HTTP. S3, GCS. Cheap, durable, scales infinitely. Presigned URLs.

Rate limit — token bucket (most popular) / fixed window / sliding window. Distributed via Redis.

CAP — under partition, pick C or A. PACELC: else, pick L or C.

Replication — master/replica for reads. Async = lag, sync = slow writes.

Sharding — split data by key. Hash / range / consistent. Hard to reshard.

Push (mobile) — APNs (iOS) / FCM (Android). Token-based. Silent for IoT background sync.

MQTT — IoT messaging. Lightweight, QoS levels, broker buffers offline clients.

OTA — A/B partitions, signed manifest, staged rollout, rollback.

OOP — encapsulation, abstraction, inheritance, polymorphism.

SOLID — Single resp / Open-closed / Liskov / Interface seg / Dep inversion.

60. How WhatsApp Works

(Part 2 §19 covered "chat app" generally. Here, the WhatsApp-specific design notes.)

Defining traits

  • 2B+ users, 100B+ messages/day
  • End-to-end encryption (Signal protocol)
  • Phone-number identity (no separate username)
  • Multi-device sync (one primary phone + companion devices)
  • Tiny payload sizes — designed for slow networks

Architecture highlights

  • Built on Erlang/OTP (now mixed with FreeBSD/C++) — supercharged for millions of concurrent connections per box
  • XMPP-style custom protocol over a long-lived TCP socket; HTTP/2 fallback
  • One ejabberd-style server box can hold ~2M concurrent connections (vs. ~100K on a typical Java stack)

End-to-end encryption (Signal protocol)

Each user has:
  Identity Key Pair       (long-lived)
  Signed Pre-Key          (rotated)
  One-Time Pre-Keys       (consumed per session)

Session setup (X3DH):
  Sender fetches recipient's keys from server (server stores only public keys)
  Derives a shared secret without server seeing private keys
  Initializes a Double Ratchet for forward secrecy

The server is just a blind relay — sees only ciphertext + metadata.

Multi-device

  • Each device has its own key pair
  • Sender encrypts to each recipient device separately
  • Primary phone still required for new-device pairing (QR code)

Status, last seen, typing

  • Ephemeral data — Redis with short TTL
  • Privacy-controlled per user

Voice/video

  • WebRTC for media
  • Signaling over the XMPP socket
  • TURN relays for NAT traversal when peer-to-peer fails

What to say in an interview

"WhatsApp pushes nearly all complexity to clients: server is a stateless relay. Erlang's lightweight processes mean one box holds millions of WS-equivalent connections — they don't shard chat the way Slack does by workspace."

61. How Reddit Works

Core entities

  • Subreddit: a community
  • Post: a submission (link / text / image / video)
  • Comment: tree-structured replies under a post
  • Vote: up/down on post or comment

Big design problems

  1. Ranking posts in a subreddit feed and comments under a post
  2. Threaded comments at huge scale (some posts have 100K+ comments)
  3. Vote fuzzing + bot resistance

Ranking algorithms (well-known and asked about)

Hot ranking (for the front page)

log10(max(|score|, 1)) * sign(score)  +  seconds_since_epoch / 45000
  • Logarithmic vote impact: 10 votes ≈ 1 unit; 100 votes ≈ 2 units (diminishing returns)
  • Time term ensures fresh posts surface; older posts decay naturally

Best (for comments) — Wilson score lower bound of the confidence interval. Resistant to small-sample noise.

Controversial — high vote count where ups ≈ downs.

Architecture

   CDN ── static → S3/CloudFront
    │
   Edge (caching) ── most pages anonymous, cache 5min
    │
   App tier (Python/Pylons historically, Go services today)
    │       │
    │       ├─→ Cassandra (posts, comments, votes)
    │       ├─→ PostgreSQL (users, subreddits)
    │       ├─→ Redis / memcached (ranking, hot lists)
    │       └─→ Kafka (events, vote streams)
    │
   Async workers: ranking recompute, anti-spam, notifications

Comments tree

  • Store comments with (post_id, parent_id, comment_id, score, created_at)
  • Fetch by post_id with secondary index on (parent_id, created_at)
  • Render lazily — fetch top N at root, expand on demand
  • Pre-rank top thread for fast initial page

Vote fuzzing

65. How Nginx Works

(System-internals question — interviewers love this for systems engineers.)

What Nginx is

A reverse proxy + web server + load balancer, designed around an event-driven, non-blocking architecture.

The classic problem it solved

Apache used a process-per-connection model: 10K connections = 10K processes = RAM explosion. Nginx handles 100K+ connections per box with single-digit processes.

Architecture

Master process    (privileged, reads config, manages workers)
   ├─ Worker 1     (single thread, event loop)
   ├─ Worker 2     (single thread, event loop)
   ├─ Worker 3     (...)
   └─ Worker N     (typically one per CPU core)

Each worker uses epoll (Linux) or kqueue (BSD/Mac) to multiplex thousands of sockets in a single thread.

Event loop (conceptual)

loop:
  events = epoll_wait(fd_set, timeout)
  for event in events:
      if event is accept: register new socket
      if event is readable: read what's available, advance state machine
      if event is writable: write what's pending
      if event is timer: handle timeout

No blocking calls. No thread-per-request. Memory per connection is tiny (~2KB).

Module / phase architecture

A request flows through phases: post_read → rewrite → access → content → log. Each module hooks into a phase. Config (location / { ... }) declares behavior per phase.

Reverse-proxy mode

upstream backend {
   server 10.0.0.1:8080;
   server 10.0.0.2:8080;
}
server {
   location / {
      proxy_pass http://backend;
   }
}

Connection pooling to upstream, keep-alive, slow-client buffering (clients can be slow; backends should never be exposed to that).

Why interviewers ask

Demonstrates understanding of:

  • Event-driven I/O vs thread-per-request
  • Worker process model
  • Why C10K (10,000 concurrent connections) was hard before Nginx

68. How Uber Computes ETA

The naive version

"Take Google Maps directions, get duration." Too slow at 1M requests/sec; not accurate enough.

Step 1: Map matching Convert (lat, lng) → nearest road segment using a precomputed road graph.

Step 2: Routing Standard graph algorithm (Contraction Hierarchies / CH for ultra-fast shortest paths).

  • Origin, destination
  • Time of day, day of week
  • Recent traffic on each segment
  • Historical traffic on each segment
  • Weather, events

Request "ETA from A to B"
   → Map match (precomputed)
   → Routing service (CH on road graph)
   → ETA service (ML model inference, latency-critical)
   → Return

All three at p99 < 100ms.

Real-time traffic

  • Driver phones constantly stream speed + position
  • Stream processor (Flink) aggregates per road segment
  • Latest segment speeds piped to a feature store
  • ETA model reads from feature store

72. How Apple AirTags Work

The problem

AirTags have no GPS, no cellular, no internet. They have only a tiny BLE radio + an accelerometer + a UWB chip. How does an AirTag in a stranger's coat pocket on the other side of the world get located?

The genius answer: crowdsourced mesh via every iPhone on Earth

1. AirTag periodically broadcasts a BLE beacon (rotating public key)
2. ANY nearby iPhone (~1B in field) hears it
3. iPhone:
     - Determines its own location (GPS)
     - Encrypts (location, time) with the AirTag's rotating public key
     - Uploads to Apple's servers
4. Owner's phone fetches encrypted reports for THEIR AirTag's keys
5. Decrypts locally (only owner has the private key)
6. Apple servers cannot see who-found-what or where

Privacy guarantees (the point of the design)

  • Apple server stores blobs it can't read
  • Helper iPhones don't know whose AirTag they reported
  • Owner phone can decrypt → location is private to them
  • AirTag's beacon key rotates frequently → can't track over time

Anti-stalking

  • AirTags emit sounds when separated from owner > N hours
  • iPhones detect "unknown AirTag traveling with me"
  • Android has companion app to scan

UWB precision finding

When you're close (~30m), UWB (ultra-wideband) takes over for cm-accurate direction.

Why this is a great interview answer

  • Demonstrates privacy-preserving system design
  • Crowdsourced compute / data model
  • Public-key crypto used cleverly (asymmetric trust)
  • BLE energy budget thinking (years on a coin-cell)

73. How Amazon S3 Works

What it is

An "infinite, durable, cheap" object store with HTTP API. Industry default.

Durability claim

"11 nines" (99.999999999%) of durability — designed to lose at most 1 in 100 billion objects per year.

How they get there

Erasure coding instead of plain replication.

  • File split into K data chunks + M parity chunks (e.g. K=10, M=4)
  • Any K of 14 → recover original
  • 1.4x storage overhead vs 3x for triple replication
  • Stored across racks, AZs, sometimes regions
Object → split → [c1][c2][c3]...[c10]  + [p1][p2][p3][p4]  (Reed-Solomon)
                  spread across 14 different disks/racks

Service architecture (per public talks)

Client → Request Routers (~99% of fleet's job is request handling)
         ├── Auth / quota / billing
         ├── Index lookup (key → location)
         └── Forward to Storage Nodes

Index: keymap service — partitioned by bucket+key prefix
Storage nodes: actual disks + erasure-coded chunks

Consistency

Until ~2020, S3 was eventually consistent. Now strongly consistent (read-after-write for new objects, updates, deletes). They achieved it without latency penalty via a coordinator-based read protocol.

Lifecycle / tiering

  • S3 Standard → Infrequent Access → Glacier → Deep Archive
  • Move automatically by lifecycle policies
  • Glacier: minutes-to-hours retrieval, 4x cheaper

S3 is a tour through erasure coding, sharded indexes, durability planning, multi-AZ resilience.

74. How Kafka Works (Internals)

(§10 covered Kafka usage. Here, internals — what makes it fast.)

Design pillars

  1. Append-only log on disk
  2. Sequential I/O beats random I/O by 100x even on SSDs
  3. Zero-copy sends to consumers (sendfile syscall, no userspace buffer)
  4. Partitioning for horizontal scale
  5. Replication for durability

Storage layout

Topic: orders
  Partition 0
    └── segment-0.log    (first 1 GB or 1 week)
    └── segment-1.log
    └── segment-N.log    (active segment - being written)
  Partition 1
  ...

Each segment is just bytes on disk: [len][crc][msg][len][crc][msg]...

Index file maps offset → byte position in log.

Producers

  • Batch messages (configurable lingering)
  • Compress per batch (snappy / lz4 / zstd)
  • Ack levels: 0 (fire-and-forget), 1 (leader acks), all (all in-sync replicas ack)

Brokers

  • Each partition has a leader broker + N replicas
  • Producers + consumers always talk to the leader
  • Followers tail the leader and ack

Consumers

  • Each consumer in a group owns some subset of partitions
  • Consumer tracks offset per partition (stored in special __consumer_offsets topic)
  • Adding a consumer triggers rebalance (partitions reassigned)

Why it's fast

  • Append-only: no btree overhead
  • Sequential disk write: hundreds of MB/s on cheap disks
  • Page cache: hot data lives in OS memory; reads bypass disk
  • Zero-copy: sendfile() from page cache → network socket, no userspace buffer

Replication protocol

  • Followers send Fetch to leader
  • Leader returns batches starting at follower's offset
  • ISR (In-Sync Replicas) = followers within N ms of leader
  • Leader fails → controller elects new leader from ISR

KRaft (the new bit)

Old Kafka used ZooKeeper for metadata. KRaft (Kafka Raft) replaces it — Kafka now manages its own metadata, simpler ops.

75. How LLMs Actually Work

The intuition

A Large Language Model is a function f(text) → next_token_distribution. Repeatedly applied → generates fluent text. Training: tweak billions of parameters so the predicted distribution matches real text.

The Transformer architecture (one paragraph)

Text is split into tokens (~subword units). Each token is mapped to a learned vector (embedding). Vectors flow through transformer layers. Each layer does:

  1. Self-attention — every token looks at every other token and pulls in context.
  2. Feed-forward — a small neural net applied independently to each token's vector. Repeated 30–100 times. Final vector → linear layer → probability distribution over the vocabulary. Sample → next token. Append. Repeat.

Why it works

  • Attention learns which earlier words matter for predicting the next one.
  • Scale (parameters, data, compute) keeps producing better results — the "scaling laws" empirical finding.

Training stages

  1. Pre-training — predict next token on trillions of words of internet/code. Teaches grammar, facts, reasoning patterns. Months of compute.
  2. Supervised fine-tuning (SFT) — train on human-written "good answers" for instructions.
  3. RLHF / RLAIF — humans (or another model) rank outputs; train a reward model; tune the policy to maximize reward. Makes it helpful + harmless.

Serving (the systems-design angle)

Request "Translate to French: Hello"
  → Tokenize  ("Translate", " to", " French", ":", " Hello")
  → Embed
  → Run through transformer (autoregressively generate tokens)
       each new token: feed back into model
  → Stop on end-token or max length
  → Detokenize → "Bonjour"

Why serving is hard

  • Each token = a full model forward pass (billions of parameters)
  • GPU memory dominated by weights + KV-cache (cached attention states for past tokens)
  • Batching across requests: dynamic batching to keep GPU busy
  • Prefill (first pass) is compute-bound; decode (per-token) is memory-bandwidth-bound — different optimizations
  • Speculative decoding: small model proposes tokens, big model verifies in parallel

What interviewers ask

  • How would you scale to 1M concurrent users? (batching, GPU sharding, KV-cache management)
  • How do you cache? (prompt prefix caching — Anthropic's prompt cache is exactly this)
  • How do you reduce cost? (quantization to int8/int4, smaller distilled models, speculative decoding)

77. How Google Docs Works (Real-time Collaboration)

The hard problem

Two users edit the same character position at the same time. How does the doc converge to the same state for everyone?

Two main approaches

Operational Transformation (OT) — Google Docs uses this. Each edit = an operation (insert at pos, delete at pos). When two ops are concurrent, server transforms each against the other so the intent is preserved.

Initial: "Hello"
Alice inserts "X" at pos 0 → "XHello"
Bob inserts "Y" at pos 5 → "HelloY"

Concurrent edits arrive at server. Server orders them. After transformation:
  Final: "XHelloY"  (both insertions land in the right place)

CRDT (Conflict-free Replicated Data Type) — used by Figma, Linear, Apple Notes. Operations are mathematically commutative — applied in any order, all peers converge.

Architecture (OT-style)

Client A ←→ Doc Server  ←→ Client B
                ↕ Client C

Each client maintains a queue of pending local ops.
Sends to server. Server orders, transforms, broadcasts back.
Client applies remote ops (transforming pending local ops against them).

Server's job

  • Single source of truth for op order per document
  • Persist ops to log (Bigtable / Spanner)
  • Snapshot the document state periodically (so replay isn't from genesis every load)

Presence + cursors

  • Each user has a "presence" record (color, cursor position)
  • WebSocket broadcasts to other collaborators
  • TTL-cleared when idle

Offline editing

  • Client buffers ops locally
  • On reconnect: send buffered ops, server transforms against any ops it received meanwhile, broadcasts result

Why interviewers ask

  • Distributed consensus at the document level (lighter than Raft)
  • Trade-off between OT (smarter server) and CRDT (smarter data type)
  • Demonstrates understanding of eventual consistency + correctness

Most chat UIs stream tokens via Server-Sent Events (SSE) so the user sees text appear immediately rather than waiting 20s.

Session memory

  • Each conversation: store messages keyed by conversation_id
  • Inference includes the message history as context
  • Long conversations: truncate, summarize, or use retrieval

Plugins / tools (function calling)

Model outputs structured "I want to call tool X with args Y". Orchestration layer:

  1. Parses tool call
  2. Executes (HTTP request, search, code run)
  3. Feeds result back to model
  4. Model continues with the result in context

Safety stack

  • Input classifier (block harmful prompts)
  • Output classifier (block harmful generations)
  • System prompt enforcement
  • RLHF-trained refusal behavior

Cost engineering

  • GPU inference is the bottleneck — $$$
  • Prompt caching for repeated system prompts (Anthropic / OpenAI both support)
  • Smaller model for routing decisions; larger for hard turns
  • Quantization (int8 weights) for cheaper inference

82. How Uber Finds Nearby Drivers at 1M Req/s

The specific problem

1 million "find drivers within R km of point P" queries per second, global, real-time. Drivers move every 4 sec.

The naive answer (and why it fails)

"PostGIS index on driver locations" → fine at small scale; collapses at this throughput.

Uber's solution (from their published talks)

1. Cell-based indexing (Google S2) Divide the world into a hierarchical grid of cells (S2 cells, ~level 12-15 ≈ 50–500 m).

Driver location → S2 cell ID → key in distributed KV
Lookup: compute S2 cells in radius around P → fetch each cell's drivers

S2 cell IDs are 64-bit integers ordered along a Hilbert curve → adjacent IDs = geographically nearby.

2. Sharding by S2 cell Each shard owns a range of S2 cell IDs. Routing a query is just hash(cell) → shard.

3. In-memory store (Ringpop / DISCO)

  • Driver positions in RAM, not DB
  • Replicated across nodes for failover
  • Updated by driver heartbeats every 4 sec

4. Decoupled write/read paths

  • Driver heartbeat → hits its "home" shard (consistent hash on driver_id)
  • Find-nearby query → hits cell-range shards
  • A separate index propagation keeps the cell-index in sync

Sketch

Driver app  --heartbeat-->  Location API  --by driver_id-->  Driver Store (KV)
                                                |
                                                +--update--> Cell Index (S2)

Rider app   --findNearby--> Match API  --by S2 cell range-->  Cell Index nodes
                                                                  |
                                                          returns driver IDs
                                                                  |
                                                                  v
                                                          Hydrate from Driver Store

Result

  • p99 < 10ms for find-nearby
  • Million-QPS sustained, with smooth scale-out by adding cells/shards

83. Real-Time Leaderboard

Requirements

  • 10M users, top-K (top 100, top 1000) at any time
  • p99 < 50ms
  • Multiple leaderboards (daily, weekly, all-time, per-region)
  • Update on every score change

Solution: Redis Sorted Sets

Redis ZSET = sorted set. Provides O(log N) insert, O(log N + K) range query.

ZADD leaderboard:daily 1500 user:42
ZADD leaderboard:daily 1700 user:43
ZRANGE leaderboard:daily 0 99 WITHSCORES REV     # top 100
ZREVRANK leaderboard:daily user:42               # user's rank

Architecture

Score event (game completion) → Kafka
                                  │
                                  v
                            Score processor
                                  │
                                  v
                        Redis cluster (sharded ZSETs)
                                  ^
                                  │
                            Leaderboard API

Sharding for huge leaderboards

A single ZSET can hold millions of users — usually fine. For tens-of-millions+:

  • Shard by user_id buckets
  • Each shard has a local top-K
  • Aggregator merges top-K from each shard

This isn't a perfect global order but is eventually-correct top-K, which is good enough for display.

Persistence

  • Redis with AOF (append-only file)
  • Periodically snapshot top-K to durable store for backup

Pitfalls

  • Hot keys: one leaderboard per region per day, not one giant global ZSET
  • Updates per second: pre-aggregate if score events flood (write to ZSET every 500ms)
  • Time-decay leaderboards: re-decay daily via background job

84. How YouTube Supports 2.49B Users on MySQL

Surprise: YouTube's core metadata DB is MySQL (via Vitess)

What's Vitess?

A YouTube-built sharding layer on top of MySQL. Now CNCF open source.

Application
     │
     v
+------------+
|  VTGate    |  ← stateless proxy; parses SQL, routes to right shard
+------------+
     │
     v
+------------+   +------------+   +------------+
|  VTTablet  |   |  VTTablet  |   |  VTTablet  |
+------------+   +------------+   +------------+
|  MySQL     |   |  MySQL     |   |  MySQL     |   ← actual shards
+------------+   +------------+   +------------+

What Vitess gives you

  • Transparent sharding: app writes regular SQL; VTGate figures out which shard
  • Online resharding: split a shard into two without downtime
  • Connection pooling: thousands of app servers → handful of MySQL connections
  • Query rewriting: catches scatter-gather queries that would hit all shards (warns or rejects)
  • Replication topology management: auto-failover on leader death

Why MySQL still?

  • Mature ops, well-known
  • ACID where YouTube needs it (subscriptions, monetization)
  • Vitess lets them scale without abandoning SQL

Shard key

  • Most tables sharded by user_id or channel_id
  • Cross-shard queries are rare — model carefully

Lessons for interviews

  • Sharding doesn't require switching to NoSQL
  • A smart proxy can hide complexity from the app
  • Vitess pattern applies to PostgreSQL too (Citus does similar)

85. How ChatGPT Apps / Plugin Platform Works

What it is

A marketplace where third parties build "apps" that the LLM can invoke. (OpenAI's Plugins / GPTs / Apps platform, Anthropic's MCP — similar idea.)

Anatomy

User chats: "Book me a flight to NYC tomorrow"
     │
     v
+--------------------+
| ChatGPT model      | → "I should use the FlightApp tool"
+--------------------+
     │
     v
+--------------------+
| Orchestration      |
| Tool registry      | → looks up FlightApp spec
+--------------------+
     │
     v
+--------------------+   HTTPS   +-----------------+
| Plugin Bridge      | --------→ | Third-party API |
+--------------------+ <-------- | (with user auth)|
     │       results              +-----------------+
     v
+--------------------+
| ChatGPT model      | → integrates result into reply
+--------------------+

Plugin manifest

Each plugin publishes an OpenAPI spec + an ai-plugin.json describing:

  • Name, description (what the model uses to decide when to invoke)
  • Authentication method (none / API key / OAuth)
  • List of endpoints with schemas

Security challenges

  • Prompt injection — a plugin returns malicious text that "tells" the model to leak user data. Mitigated by sandboxing tool outputs.
  • Cross-tool data leakage — plugin A sees data from plugin B's response. Mitigated by per-tool isolation.
  • Auth — user-scoped OAuth tokens stored encrypted, never exposed to other plugins.

Performance

  • Tool call adds 1–3 seconds (extra round trip + extra LLM pass)
  • Tool result counts against context window — can't dump 100KB into the LLM
  • Many platforms now support parallel tool calls to reduce latency

Anthropic's MCP (Model Context Protocol)

A standard protocol for tools to expose themselves to any LLM. Decouples tools from any one model vendor.

Why interviewers ask

  • Multi-actor orchestration
  • API gateway + auth + sandboxing under LLM constraints
  • Forces candidate to think about security with non-deterministic clients
