Handbook · Interview Guide

System Design, from HTTP to distributed systems.

A working reference for interview-oriented system design: foundational building blocks, the classic problems interviewers ask, IoT/mobile-specific designs, low-level design, and a playbook for the 45-minute round. Each section aims to be short enough to skim, deep enough to teach the trade-offs.

Part 1 — Foundations

01What system design is (and what interviewers score)

System design is the practice of turning a vague product ask ("design Instagram") into a concrete plan: what the system must do, how well it must do it, which components handle each job, and how those components survive real failure.

Interviewers score four things, roughly in this order:

  • Requirements clarification — do you pin down scale, read/write ratio, and consistency needs before drawing boxes?
  • Structured approach — clarify → estimate → high-level → deep-dive → failures, not a random box tour.
  • Trade-off awareness — every choice has a cost. Naming the cost is the point.
  • Depth — you go deep on at least one component (schema, algorithm, failure recovery), not shallow across many.
The five-step frame
Clarify · Estimate · High-level · Deep-dive · Scale & Fail. This shape covers 95% of prompts.

02Client–server model

A client asks for data — phone, browser, IoT device, another server. A server waits for requests and sends responses. They speak a shared protocol over a network — usually HTTP.

Every system-design question ultimately reduces to "how do clients get data to/from servers, and what happens server-side?" Even "design Uber" is two clients (rider, driver) talking to a coordinated server backend.

03HTTP, REST, and APIs

HTTP is a request–response protocol. Client sends a request; server sends one response.

GET /users/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbG...

Verbs

Idempotency matters for retries: if a network blip causes a client to retry, you don't want a duplicate charge.

Status codes worth memorizing

2xx success (200, 201, 204). 3xx redirects (301). 4xx client mistake (400 bad request, 401 unauthenticated, 403 unauthorized, 404 not found, 409 conflict, 429 too many). 5xx server fault (500, 502, 503, 504).

REST style

Model your API around resources (nouns), not RPC verbs. Use plural nouns, HTTP methods for action, version in the path, and cursor-based pagination for long lists.

POST   /v1/users        create
GET    /v1/users/42     read
PATCH  /v1/users/42     update
DELETE /v1/users/42     delete
GET    /v1/users/42/orders?limit=20&cursor=abc

When request/response isn't enough — WebSocket

HTTP is one-way (client initiates). For server-push (chat, live dashboards, games) you want a persistent bidirectional pipe. WebSocket upgrades an HTTP connection into that pipe. Alternatives: Server-Sent Events (one-way push), long-poll (a hack).

04DNS — how a request reaches a server

1. App → OS resolver:    "api.example.com?"
2. Resolver → Root DNS:  "ask .com"
3. Resolver → .com:      "ask example.com"
4. Resolver → example:   "52.84.123.45"
5. App opens TCP → TLS → HTTP

DNS is cached at multiple layers (OS, browser, resolver). GeoDNS routes based on client region — the foundation of CDN routing. It's also a real single-point-of-failure risk; large sites use multi-provider DNS.

05Load balancers (L4 vs L7)

One server handles maybe 1,000 req/s. You need 100,000. So 100 servers. But clients only know one address — the load balancer, which distributes traffic across the backend pool.

Algorithms

  • Round-robin — cycle through servers.
  • Least connections — send to whoever has the fewest active connections.
  • Weighted — larger boxes get more.
  • IP hash — same client → same server (for sticky sessions).

L4 vs L7

L4 (transport) routes on IP + port, doesn't understand HTTP — fast but dumb. L7 (application) understands HTTP, can route on path/headers/cookies — smarter but slower. Modern LBs: AWS ALB/NLB, NGINX, HAProxy, Envoy.

Health checks

LB pings each backend (GET /health). If N consecutive checks fail, remove from pool. When it recovers, add back. The LB itself needs HA — run active/active pairs with a floating VIP, or use a managed cloud LB.

06API gateways

An API gateway is the smart front door for your system. It sits at the edge and handles cross-cutting concerns so backend services don't have to:

  • Routing (/users → user service, /payments → payments)
  • Auth (validate JWT before forwarding)
  • Rate limiting
  • Request/response transformation (schema v1 → v2)
  • TLS termination, logging, metrics

Load balancer vs gateway: an LB spreads traffic across identical backends. A gateway routes to different backends and adds smarts. Examples: AWS API Gateway, Kong, Apigee, Envoy, Spring Cloud Gateway.

07Databases — SQL vs NoSQL

SQL (relational)

Rows and columns, foreign keys, joins, ACID transactions. Mature, expressive, well-understood. Downside: sharding is painful. Use for: transactions, structured data with relationships (users, orders, payments). Examples: PostgreSQL, MySQL, Oracle.

NoSQL — four flavors

Picking a DB in an interview

Start from the query pattern, not the buzz. If you'll do multi-row transactions and joins, SQL. If write throughput and simple key access dominate, wide-column. If you need session state / cache, KV. If your primary axis is time, time-series.

08Indexes, transactions, ACID, BASE

Indexes

Like a book's index: instead of scanning every row, the DB jumps to matching rows via a data structure — typically a B-tree (range-friendly) or hash (equality only). Trade-off: reads faster, writes slower (every insert must also update the index). Composite indexes ((user_id, created_at)) — column order matters for which queries can use them.

ACID

  • Atomicity — all-or-nothing.
  • Consistency — DB constraints are honored across a transaction boundary.
  • Isolation — concurrent transactions don't see each other's half-done work.
  • Durability — once committed, changes survive a crash.

Isolation levels

Read Uncommitted → Read Committed → Repeatable Read → Serializable. Higher = stricter but slower. PostgreSQL defaults to Read Committed; MySQL InnoDB defaults to Repeatable Read.

BASE

The NoSQL counterpoint: Basically Available, Soft state, Eventual consistency. Don't expect ACID guarantees; expect that the system stays up and converges eventually.

09Caching

RAM is ~100× faster than a DB round-trip. Real systems get 10–1000× speedups from caching. Caches live at every layer: browser, CDN edge, API gateway, in-process, distributed (Redis/Memcached), DB internal, disk.

Write strategies

  • Write-through — write to cache AND DB synchronously. Cache always fresh; slow writes.
  • Write-behind — write to cache, ack client, flush to DB async. Fast writes; data loss risk if cache dies.
  • Write-around — writes bypass cache and go straight to DB; cache is populated on first read miss.

Read strategies

Cache-aside (lazy) — the most common pattern:

def get_user(id):
    u = cache.get(f"user:{id}")
    if u is None:
        u = db.get(id)
        cache.set(f"user:{id}", u, ttl=300)
    return u

Eviction

LRU (least recently used), LFU (least frequently used), FIFO, TTL. LRU is the default for a reason — it approximates locality well.

10Message queues & event-driven architecture

Service A wants to tell B "user signed up — send welcome email." Direct call couples them: A is slow if B is slow, and A breaks if B is down. A queue between them decouples:

A drops a message and moves on. B reads when ready. If B is down, messages pile up safely.

Queue vs Pub/Sub

  • Queue (point-to-point) — each message goes to one consumer. SQS, RabbitMQ. Good for task distribution.
  • Pub/Sub (broadcast) — each message goes to every subscriber. Kafka, SNS, Google Pub/Sub. Good for events.

Kafka in one paragraph

Kafka is a distributed append-only log. Topics have partitions; each partition is an ordered sequence of messages retained for days/weeks. Multiple consumer groups read independently, each tracking its own offset. Ordering is guaranteed within a partition. This is what makes it a great fit for event sourcing, streaming, and fan-out to many independent consumers.

Delivery guarantees

At-most-once may drop messages. At-least-once may duplicate — the common default. Exactly-once is hard; usually achieved via consumer-side idempotency (dedup key). Design consumers to be idempotent.

Patterns

Fan-out (one event, N consumers). Worker pool (many workers pull from one queue). Event sourcing (store every state change as an event; rebuild state by replay). CQRS (separate write and read models). Saga (multi-step distributed transaction with compensating actions).

11CDNs

A network of edge servers around the world caching static content close to users. A user in Mumbai fetches from a Mumbai edge in ~5ms instead of a US origin in ~200ms.

What goes in a CDN: images, video, JS, CSS, fonts, downloads, sometimes short-TTL API responses. Invalidation via TTL, explicit purge, or cache-busting filenames (app.a3f8b9.js). Examples: CloudFront, Cloudflare, Fastly, Akamai.

12Object storage (S3-style)

An HTTP service for blobs — cheap, massively durable, effectively infinite. Buckets contain objects addressed by key. Versioning, lifecycle policies (auto-tier hot → cold → archive), and presigned URLs for direct upload/download without exposing credentials.

Why not put big blobs in a DB? Databases charge dearly per GB and choke on multi-MB rows. Object storage is designed for "write once, read many." Use for: user uploads, backups, static hosting, data lakes, firmware images.

13Rate limiting

Why

Prevent abuse and DoS, fairly share among tenants, protect downstream services from being overrun.

Algorithms

  • Fixed window — counter per minute. Simple; suffers a 2× spike at boundaries.
  • Sliding window log — keep timestamps of recent requests. Precise; memory-heavy.
  • Sliding window counter — weighted blend of current + previous window. Low memory, smooth, approximate.
  • Token bucket — bucket refills at rate R up to capacity C; each request costs 1 token. Allows short bursts, smooths sustained rate. The most popular.
  • Leaky bucket — queue drains at constant rate; overflow drops. Enforces smooth output.

Distributed

Counter must be shared across all gateway instances. Common: Redis with INCR + EXPIRE, or a Lua script for atomic token-bucket refill. Reject with HTTP 429 and a Retry-After header.

14Authentication & authorization

AuthN = who are you? AuthZ = what may you do?

Session-based

Server issues a session ID, stores server-side (Redis/DB), client carries it in a cookie. Easy to revoke by deleting the session row. Needs shared state across servers.

Token-based (JWT)

Server issues a signed token containing claims. Client sends it on every request; server verifies signature — no DB lookup. Stateless, scales easily. Harder to revoke before expiry — mitigate with short-lived access tokens + longer refresh tokens.

OAuth 2.0

"Log in with Google." Your app never sees the user's Google password. Flow: redirect user to Google → user approves → Google redirects back with a code → your backend exchanges code + secret for an access token → use token to call Google APIs.

Authorization models

RBAC (roles → permissions), ABAC (rules over attributes: "editor if user.id == doc.owner_id"), ACL (explicit per-resource grants).

15Scaling — vertical, horizontal, replication, sharding

Vertical (bigger box) is easy but capped. Horizontal (more boxes) scales further but adds coordination cost. Aim for stateless app servers — any request can go to any server, so you can just add boxes.

Replication (read scaling)

One primary handles writes; N replicas serve reads. Replicas lag behind by ms–seconds. Reads that must see the freshest data go to the primary; others can go to replicas.

Sharding (write & storage scaling)

Split data across N DBs by a shard key. Options:

  • Range sharding — A–F → shard 1, G–M → shard 2. Simple but risks hot ranges.
  • Hash sharding — hash(key) % N. Uniform; painful to add nodes (nearly everything remaps).
  • Consistent hashing — keys and nodes on a ring; adding a node moves only ~1/N of keys. Used by Cassandra, DynamoDB.

Cross-shard queries are painful. Design shard keys around your hottest queries. Never use monotonically increasing values (created_at) as a shard key — creates a hot shard.

16Consistency, availability, CAP, PACELC

CAP: under a network partition, a distributed system must choose Consistency or Availability. Since partitions will happen, the real question is CP or AP. Cassandra and DynamoDB are AP by default (tunable). ZooKeeper, etcd, and MongoDB (default) are CP.

PACELC is the smarter version: if Partition, choose A or C; Else, choose Latency or Consistency. Even with no partition, replication trades latency for consistency — do you wait for all replicas to ack, or return fast and let them catch up?

Consistency spectrum

Eventual → read-your-writes → monotonic reads → causal → linearizable. Pick the weakest that your feature can tolerate — stronger consistency costs latency and availability.

17Observability — logs, metrics, traces

Three pillars, three answers to different questions.

  • Logs — discrete events. "What happened at 14:32?" Stacks: ELK/Opensearch, Loki, Datadog.
  • Metrics — aggregate numbers over time. "How's request rate?" Stacks: Prometheus + Grafana, Datadog.
  • Traces — request paths across services. "Where's the latency?" Stacks: Jaeger, Zipkin, OpenTelemetry.

Alerts sit on top of metrics — page on-call when a rule fires (p99 latency > 500ms for 5 min, error rate > 1% for 1 min). In interviews, land the plane with: "I'd export RED metrics (Rate/Errors/Duration) per service via OpenTelemetry, dashboard in Grafana, alert on p99 and error rate."

Part 2 — Classic Problems

Each of these follows the same structure: requirements → estimates → high-level → deep-dive → scale & failure.

18URL Shortener

Requirements

Shorten a long URL to a short one; redirect on read; optional custom alias, expiry, analytics. Assume 100M writes/day → ~1,200 writes/sec; 10:1 read:write → ~12K reads/sec; latency p99 < 100ms.

High-level

Generating short codes

Three approaches:

  1. Hash the URL — deterministic, but collisions need handling.
  2. Counter + base62 — no collisions, but a global counter is a bottleneck. Fix by pre-allocating ID batches to each app server.
  3. Random + collision check — simple, one extra DB lookup per write.

Prefer counter with batch allocation.

Read path

GET /aB3xZ9
  cache.get → hit → 302 to long URL
  miss → db.get → cache.set → 302

Failure & scale

Redis cluster for hot URLs. DB sharded by short_code. CDN cache the redirect endpoint with a short TTL. Analytics goes through Kafka — never write per click to the DB directly, aggregate hourly.

19Chat App (WhatsApp / Slack)

Requirements

1:1 and group messages, delivery + read receipts, offline delivery, media, message history, presence. Assume 500M users, 50M concurrent, p99 send-to-receive < 500ms.

High-level

Connection management

50M WebSocket connections → hundreds of gateway boxes. Route users to gateways via consistent hashing on user_id. A presence service (Redis hash) maps user_id → gateway_id.

Message flow

  1. Alice's message hits her gateway.
  2. Gateway persists (via queue or direct DB) and acks Alice.
  3. Look up Bob's gateway in presence.
  4. Forward to Bob's gateway over internal RPC.
  5. Bob's gateway writes to Bob's WebSocket. Bob's app renders + sends "delivered".

Storage

Cassandra is a natural fit: partition by conversation_id, cluster by message_id (TIMEUUID desc). Range scan the latest N messages in a partition is O(1) seeks. For 1:1 chats, define conversation_id = min(uid_a, uid_b) + ":" + max(uid_a, uid_b).

Groups

Store the message once per conversation; each user maintains a last_read pointer. Don't fan out to individual inboxes for groups.

Offline & push

If recipient is offline, persist and push via APNs/FCM. When they come online, pull messages since last_seen_message_id.

End-to-end encryption

Server becomes a blind relay — messages encrypted client-side (Signal protocol). Group chats use "Sender Keys" (one symmetric key per group, distributed via pairwise encryption).

20Notification System

Requirements

Send push/email/SMS/in-app. Templates. Per-user preferences per channel. Retries, dedup. Priority lanes (transactional vs marketing).

Architecture

Deep dives

  • Idempotency — each notification carries an ID; workers check for prior send.
  • Retries — exponential backoff; cap attempts; dead-letter queue for terminal failures.
  • Per-provider rate limits — token bucket per provider per app.
  • Dedup window — same user + template + data within 5 min → drop.
  • Priority lanes — separate Kafka topics for OTP vs marketing; different SLOs and worker pools.
  • Marketing blast — spread a 10M-user broadcast over 5 min to avoid hammering APNs.

21Ride Booking (Uber)

Pieces

Riders, drivers, trips, real-time location, ETA, fare, payments, surge.

Location indexing

You need "drivers within R km of point P" fast. Options: geohash (encode lat/lng as a string, prefix match = nearby), quadtree, S2 (Google, maps Earth to a 1D Hilbert curve), or Redis GEOADD/GEOSEARCH.

Matching

Query available drivers within R of pickup. Score by distance × ETA × rating × surge. Offer to top driver with a 10s window; on decline/timeout, go to next. On accept, create the trip.

Driver heartbeat

GEOADD drivers:available -122.41 37.77 driver_123
EXPIRE driver:123 10

TTL means stale drivers vanish naturally.

Trip state machine

REQUESTED → MATCHING → ACCEPTED → ARRIVED → IN_PROGRESS → COMPLETED / CANCELLED

Each transition emits a Kafka event; other services (notify, charge, analytics) subscribe.

Surge

Per-region ratio of requests to available drivers, recomputed every 1–2 min.

22Food Delivery

Three-sided match: customer ↔ restaurant ↔ driver. Driver is assigned time-windowed (5 min before food is ready), not up front. Search index: ElasticSearch on cuisine, location, rating. Order pipeline is a Saga: payment → kitchen ack → driver dispatch → delivery, with compensations on failure. ETA prediction blends restaurant prep-time history + traffic + driver location.

23Instagram Feed

Two approaches

  • Pull (read fan-out) — at feed load, query recent posts from users I follow, merge sort. Cheap writes; slow reads if I follow many.
  • Push (write fan-out) — on post, write to every follower's inbox. Fast reads; expensive for celebrities (100M writes).

Hybrid (what real systems do)

Push for normal users (< ~10K followers). Pull for celebrities — merge celebrity posts at read time.

Architecture

POST /posts → save → Kafka
                            ↓ fan-out worker
                       Redis lpush follower:feed post_id (if normal)
                       mark as celebrity-post (if huge)
GET  /feed → Redis lrange → merge celebs → hydrate → return

Ranking is a separate concern — modern feeds are ML-ranked by engagement prediction, not chronological.

24File Sync (Dropbox / Google Drive)

Key ideas

  • Chunking — split files into ~4MB chunks; upload in parallel; resumable.
  • Content-addressed dedup — hash each chunk (SHA-256). If the server already has that hash, skip the upload. Massive storage savings.

Upload flow

  1. Client chunks + hashes locally.
  2. Client POSTs metadata + hash list.
  3. Server responds "I have chunks A, C; send B, D, E."
  4. Client uploads missing chunks via presigned S3 URLs.
  5. Server commits file version.

Multi-device sync

Each device has a local DB of (file_id, version). Poll or WebSocket for changes; server returns the delta. Conflicts: last-write-wins for simple cases, "conflicted copy" file for competing writes.

25Video Streaming (YouTube / Netflix)

Upload + transcode

Upload → S3 raw → Kafka → transcode workers (240p/480p/720p/1080p/4k)
       → HLS segments (~6s .ts files) + manifest (.m3u8) → CDN
       → DB (metadata: status = ready)

Delivery — adaptive bitrate

Video pre-encoded at multiple bitrates. Manifest lists them all. Player measures bandwidth and picks a bitrate per segment. CDN is non-negotiable — origin should almost never be hit for popular content.

Netflix-style pre-positioning

Less popular titles aren't on every edge. Push to regional caches before launch (Open Connect appliances inside ISPs).

26Payment System Basics

Non-negotiables

  • Idempotency — client sends an Idempotency-Key; duplicate requests return the prior response. Never charge twice.
  • Atomicity — balance debit + ledger entry commit together in one DB transaction.
  • Audit log — every state change is an append-only event.
  • Reconciliation — nightly job matches your ledger against bank statements.

Double-entry ledger

Every transaction is two entries. Transfer ₹500 from Alice to Bob:

ledger: (alice, -500)
ledger: (bob,   +500)
sum of all entries = 0 always

Never store raw card numbers

Send them to the payment gateway (Stripe/Adyen) and store only the gateway's token. This keeps you out of PCI DSS scope.

27Typeahead / Search Autocomplete

Data structure

A trie — each node stores a character; terminal nodes store popularity. At each node, pre-compute the top-K descendants (top 5 by popularity) so we don't traverse on the hot path.

Architecture

Client → API → in-memory trie (per server)
                    ↑
       nightly load from offline pipeline:
       aggregate 7d search logs → time-decayed popularity → build trie → S3 snapshot

Personalization

Small per-user trie of recent queries + global trie. Merge at read time.

28Distributed Rate Limiter

N gateway instances need to enforce a shared limit. Token bucket in Redis via a Lua script (atomic refill + take). Sharded counters (by user_id hash) spread load. Local per-instance caches with periodic Redis reconciliation reduce Redis QPS at the cost of slight over-allowance. On Redis failure: fail open with a local safety cap and a circuit breaker.

29Logging & Monitoring

App → structured JSON logs → agent (Fluent Bit / OTel)
                            → Kafka → indexer → Elastic/Opensearch → Kibana

App → /metrics → Prometheus → Grafana / alerts

Concerns at scale: volume (compress, sample, hot/cold tiers), retention (7d hot / 90d warm / 1y cold in S3), PII (redact at the agent), cardinality (never put user_id as a metric label — it explodes).

Part 3 — IoT / Mobile-Specific

30Mobile App Backend

Mobile backends differ from web in ways worth naming:

  • Flaky networks and high latency — retries, offline caches.
  • Battery and bandwidth matter — small payloads, compression.
  • Push notifications instead of polling.
  • App versioning — old clients live forever; you can't force update.

Best practices: versioned APIs (/v1, /v2); feature flags; a /config endpoint that returns min supported version so clients show upgrade screens; delta sync (client sends last sync token, server returns only changes); Protobuf for hot paths.

31Multi-Device Sync

Conflict resolution

  • Last-write-wins — highest timestamp wins. Works for notes, settings.
  • CRDTs — data types that mathematically converge under any merge order. Used by Figma, Linear, Apple Notes.
  • Operational Transformation — each edit is an operation; concurrent ops are transformed against each other. Used by Google Docs.

Sync protocol

GET /sync?since=<token> → server returns diff + new token
POST /sync with local_ops → server acks + new token

Every item carries (id, value, modified_at, modified_by_device, deleted_flag). Soft-delete via deleted_flag; never hard-delete until tombstone TTL.

Push over poll

Silent push (APNs/FCM) wakes the app, which then runs a /sync. Way better on battery than polling.

32Offline-First App

Four rules:

  1. Local DB is the source of truth for the UI.
  2. Mutations write to local DB first, then to an outbox for sync.
  3. Outbox is durable across restarts.
  4. UI shows optimistic state immediately; reconciles on server ack.

Battery-friendly sync triggers: app foreground, significant location change, silent push, network changed cellular → Wi-Fi.

33Push Notifications (Deep Dive)

iOS — APNs

Server holds a .p8 auth key. Device gets a device token from APNs at app launch and uploads it. Server sends to APNs's HTTP/2 endpoint with {device_token, payload}.

Android — FCM

Same shape with FCM tokens and the FCM Admin SDK.

Constraints

APNs and FCM: 4KB payload max. For rich content, iOS uses a Notification Service Extension to download images after delivery.

Token lifecycle

Tokens rotate on uninstall/reinstall/device restore. Server must dedup per user. Treat 410 Gone / Unregistered as terminal — delete the token.

Silent notifications

iOS content-available: 1 wakes the app briefly with no UI — the standard mechanism for background sync.

34BLE / Wi-Fi Device Setup

Common flow for a headless smart device (kettle, scale, thermostat):

  1. Device boots into setup mode → advertises BLE.
  2. App scans, finds it, connects (encrypted with ECDH).
  3. App reads visible SSIDs from device.
  4. User picks SSID + enters password in the app.
  5. App sends Wi-Fi creds over BLE.
  6. Device disconnects BLE, joins Wi-Fi, calls /claim.
  7. App confirms ownership via a one-time code read over BLE.

Security musts

No plaintext credentials over BLE. Device certificates pre-provisioned at factory or JITP via AWS IoT. Replay protection via nonces. Claim flow proves ownership to the cloud.

State machine

UNCLAIMED → SETUP_MODE → WIFI_CONNECTING → CLOUD_CONNECTING → CLAIMED → ONLINE

35IoT Telemetry Pipeline

Why MQTT for IoT

Designed for unreliable networks. Persistent sessions (broker buffers while device offline). QoS levels (0 fire-and-forget, 1 at-least-once, 2 exactly-once). Tiny headers.

Backpressure

MQTT broker buffers; Kafka absorbs spikes with days of retention; consumers scale horizontally per partition. Devices buffer locally on disconnect and flush on reconnect (small ring buffer in flash).

Cost knobs

Sample rate, aggregation tiers (raw 30d → 1-min avg forever), compression (Parquet + zstd for cold).

36Time-Series Data Storage

Time-series DBs (InfluxDB, Timescale, Prometheus) are optimized for append-only writes and range-over-time + group-by-tag queries. Internals:

  • Chunks by time — a "hypertable" is partitioned by time (e.g. 1-day chunks).
  • Columnar storage — timestamps and values stored in columns, great compression.
  • Delta / delta-of-delta encoding — timestamps often at regular intervals, so store the diff of diffs. Facebook's Gorilla gets ~1.4 bytes/sample.
  • Retention policies — drop chunks older than N days automatically.

Don't use a TSDB for heavy updates or high-cardinality dimensions (like user_id as a tag — cardinality explodes).

37OTA Firmware Update

Requirements

Push new firmware to a fleet safely. Staged rollout (1% → 10% → 100%). Rollback on failure. No bricks. Signed images.

Device flow

  1. Device polls /firmware/check?fw=&hw=&serial=.
  2. Server applies rollout rules → returns a manifest or "no update."
  3. Device verifies manifest signature.
  4. Downloads image in chunks (resumable). Verifies SHA-256 + image signature.
  5. Writes to inactive partition (A/B partitions).
  6. Sets boot flag → reboot.
  7. New firmware runs a self-check; if OK, marks itself confirmed. If it fails to check in within N reboots, bootloader rolls back to the previous partition.

Staged rollout

Bucket each device by hash(serial) % 100. Server says "give me 2.4.0 only if bucket < 10." Watch crash and return metrics; expand bucket.

Why A/B partitions

Atomic switch (change boot pointer). Rollback = boot the other partition. Never overwrite the running firmware.

38Device Provisioning

Lifecycle: factory-provisioned (serial + unique key + cert) → shipped → first-time onboarding → claimed → operational → decommissioned. The claim flow proves ownership: device signs an identity challenge with its factory key; app confirms with a one-time code read over BLE. Multi-tenant isolation via mTLS with per-device certs and per-device IAM policies.

Part 4 — Low-Level Design (LLD)

39OOP Essentials

An object bundles state (fields) with behavior (methods). The four pillars:

  • Encapsulation — hide internals behind methods.
  • Abstraction — expose what, hide how.
  • Inheritance — extend a class. Use sparingly; prefer composition.
  • Polymorphism — same interface, different behavior.

UML shorthand for interviews

- private   + public   # protected
◇── composition   ◆── aggregation
──▷ inheritance   ---▶ association

40SOLID

  • S — Single Responsibility. One reason to change per class.
  • O — Open/Closed. Open for extension, closed for modification. Add a new payment type by implementing an interface, not editing a switch.
  • L — Liskov Substitution. Subtypes must work anywhere the base type does. The classic Square extends Rectangle pitfall violates this.
  • I — Interface Segregation. Don't force clients to depend on methods they don't use. Prefer many small interfaces.
  • D — Dependency Inversion. Depend on abstractions, not concretions. Inject dependencies rather than new-ing them inside a class.

41Design Patterns Worth Knowing

Creational

Singleton — one instance (use sparingly, prefer DI). Factory — encapsulate object creation. Builder — step-by-step construction. Prototype — clone an existing instance.

Structural

Adapter — make an incompatible interface fit. Decorator — wrap to add behavior. Facade — single front door over a complex subsystem. Proxy — stand-in (lazy load, access control, remote). Composite — tree structures where leaves and containers share an interface.

Behavioral

Observer — subject notifies subscribers on change. Strategy — interchangeable algorithms via an interface. State — behavior varies with internal state, replacing large switch statements. Command — wrap an action as an object (enables undo, queueing). Iterator — uniform traversal. Template Method — base class defines skeleton, subclasses fill steps. Chain of Responsibility — request travels a chain until one handler handles it (middleware).

42Dependency Injection & Architectures

DI

Don't construct dependencies inside a class; pass them in. Enables testing with fakes:

class OrderService:
    def __init__(self, repo: OrderRepo, mailer: Mailer):
        self.repo = repo
        self.mailer = mailer

MVC / MVVM / Clean

  • MVC — Model, View, Controller. View → Controller → Model → View.
  • MVVM — ViewModel exposes observable state; View binds to it. Testable without UI. iOS/SwiftUI default.
  • Clean Architecture — layers with dependencies pointing inward: UI → adapters → use cases → entities. Use cases depend on interfaces the outer layers implement.

43Thread Safety

Terms: race condition, critical section, mutex, semaphore, atomic, RWLock, lock-free.

Patterns: bounded producer-consumer queue; reader-writer split (many readers, one writer); immutability (immutable objects are free to share). Pitfalls: deadlock (fix: acquire locks in a global order), livelock, starvation, false sharing.

44LLD Problem Catalog (Short Sketches)

Parking Lot

Classes: ParkingLot owning floors, Floor owning Slots, Vehicle hierarchy (motorcycle/car/truck), Ticket, FeeCalculator. Slot types (S/M/L) determine which vehicles fit. FeeCalculator is a Strategy (hourly, flat, peak). findAvailableSlot is O(1) with a per-floor Queue<Slot> per type.

Elevator

ElevatorSystem, Elevator, Scheduler (Strategy — nearest idle, or LOOK). Each Elevator has state (idle/up/down/doors_open) and a request queue. The LOOK algorithm — go all the way in one direction picking up consistent-direction requests, then reverse — is the interview-friendly answer.

Splitwise

User, Group, Expense with SplitStrategy (equal / exact / percentage). Balance sheet: Map<User, Map<User, Decimal>>. Settling with minimum transactions is NP-hard in general; a greedy "biggest creditor pays biggest debtor" gives near-optimal.

Chess

Game, Board, Piece abstract with each type overriding validMoves. Move as a Command (enables undo, history). Depth to score: castling, en-passant, promotion, check/checkmate detection.

ATM

Great fit for the State pattern. ATMSession transitions Idle → CardInserted → PinEntered → Authenticated → TransactionInProgress. Each state handles inputs differently. CashDispenser greedy-dispenses from largest denomination.

Logger

Logger with a list of LogAppenders (Console, File, Network). LogFormatter Strategy. Async logging via a Decorator: background thread + queue + drain. Must be thread-safe.

File System

INode abstract with File and Directory subclasses. Directory holds Map<String, INode> — pure Composite pattern. size() on Directory recursively sums children.

Vending Machine

State pattern (NoSelection → ItemSelected → HasMoney → Dispensing). Coin manager handles change like the ATM's cash dispenser.

Rate Limiter (LLD)

RateLimiter interface with allow(key): bool. Token bucket per key, thread-safe via atomics or per-key locks. Fixed and sliding window as alternate implementations.

Library Management

Distinguish Book (logical title) from BookCopy (physical instance). Loan ties member to copy with due date. Fine calculation is a Strategy.

45The 45-Minute System-Design Playbook

0:00–0:05  CLARIFY
   Features? Read-heavy or write-heavy? DAU? Peak QPS?
   Latency target? Consistency needs? Mobile + web + geos?
   Get a nod on the agreed scope before drawing anything.

0:05–0:10  ESTIMATE
   QPS = DAU × requests-per-user-per-day / 86400
   Storage = items × size × retention
   State assumptions clearly; round generously.

0:10–0:20  HIGH-LEVEL
   Draw client, gateway/LB, services, DBs, cache, queue.
   Trace the main use case through the boxes.
   Name techs with brief justification.
   Ask: "Anything in particular you'd like me to focus on?"

0:20–0:35  DEEP-DIVE
   Pick 1–2 components. Schema with columns.
   Sharding key, replication, indexes.
   A concrete algorithm. A specific failure + recovery.

0:35–0:42  SCALE & FAIL
   10× the load — what breaks first?
   Primary DB dies — how do we recover?
   Region outage — do we go down?

0:42–0:45  WRAP
   Recap trade-offs. Name 1–2 things you'd improve.
   Invite follow-up.

Phrases that score

  • "I'm assuming X for now — flag if you want me to dig in."
  • "Trade-off is Y vs Z. I'd pick Y because…"
  • "If we needed exactly-once instead of at-least-once, I'd change this to…"

Phrases that lose points

  • "I'd just use MongoDB." (no justification)
  • Long silences — think out loud.
  • Drawing 12 boxes and never going deep on any of them.

46The 45-Minute LLD Playbook

0:00–0:05  Clarify scope + main entities + interaction surface.
0:05–0:10  Identify entities from the nouns of the prompt. Group into classes; note relationships.
0:10–0:20  Class diagram. Fields + methods. Mark interfaces / abstract. Show inheritance and composition. Call out 1–2 SOLID applications.
0:20–0:35  Code the 2–3 most interesting methods for real. Handle edges. Discuss thread safety if relevant.
0:35–0:42  Extend — "how would you add feature X?" Show your design accepts change without rewrites.
0:42–0:45  Tests + wrap.

Patterns that answer ~80% of LLD prompts: Strategy (anything "calculate X / send via Y / rank by Z"), State (order, ride, payment, vending), Observer (notifications, dashboards), Factory (create one of many subtypes), Composite (folders, menus, expression trees), Decorator (logging, retry, encryption wrappers), Command (undo, queueable ops).

47Behavioral Stories

Use STAR: Situation, Task, Action, Result. Prepare 4–6 stories you can adapt to any leadership principle:

  • Simplifying a complex thing.
  • Owning an outcome end-to-end.
  • Disagreeing with a senior and how it resolved.
  • Shipping on a tight deadline.
  • Making a mistake and recovering.
  • Mentoring or cross-functional collaboration.

Write each as: one-paragraph setup, one-paragraph action (specifics — techs, numbers, stakeholders), one-paragraph result (metrics if possible: "reduced setup failures from 18% to 3%").

Part 6 — Case Studies (System Design Pyramid)

50WhatsApp

Two billion users, 100B+ messages/day, end-to-end encrypted. Built on Erlang/OTP — one box holds ~2M concurrent connections vs. ~100K on a typical stack. Custom XMPP-like protocol over a long-lived TCP socket. Signal protocol for E2E: session setup via X3DH, ongoing messages via Double Ratchet. Server is a blind relay — sees only ciphertext + metadata. Multi-device: each device has its own key pair; senders encrypt to each recipient device. Voice/video via WebRTC with TURN relays for NAT traversal.

51Twitter Timeline

Historical architecture: fan-out on write. On tweet, worker pushes the tweet ID to every follower's Redis list. Timeline read = one LRANGE + hydrate. Doesn't work for celebrities with 100M followers — one tweet becomes 100M writes. Hybrid solution: fan-out for normal users, mark celebrities so their tweets are merged in at read time. Tweets stored in Manhattan (Twitter's KV) keyed by tweet_id. Snowflake IDs give chronological order for free — 64 bits of (timestamp | machine | sequence).

52YouTube

Metadata on Vitess (sharded MySQL). Views/likes counters in Bigtable. Videos on cold blob storage, delivered via multi-CDN. Watch experience uses HLS/DASH adaptive bitrate. Recommendations use a two-tower model — one embedding for videos (title/captions/watch signal), one for users (history/context), cosine-similarity for candidate generation, then a deeper re-ranker on the top 500. Live streaming: RTMP ingest → cloud transcode → HLS segments → CDN, ~5–30s end-to-end latency.

53Uber ETA

Three stages, all under p99 100ms. Map matching — snap raw (lat, lng) to the nearest road segment. Routing — shortest path via Contraction Hierarchies on the road graph. ETA prediction — a learned model (gradient-boosted tree, then DeepETA transformer) that ingests time of day, recent traffic on each segment, historical traffic, weather, events, and the driver's current speed, and outputs predicted seconds. Real-time traffic comes from driver-phone speeds streamed into a feature store.

54Uber: Finding Nearby Drivers at 1M req/s

Cell-based indexing with Google S2 — the Earth is a hierarchical grid; each cell has a 64-bit ID ordered along a Hilbert curve so adjacent IDs are geographically nearby. Shard by cell-ID range. Driver positions live in RAM (Ringpop / DISCO), not in a DB — heartbeats every 4s update the home shard. Find-nearby queries hit the cell-index nodes for the S2 cells covering the search radius. p99 < 10ms.

55Airbnb

Search + ranking + bookings + messaging in one product. Search: ElasticSearch on geo_point, price, amenities, per-listing availability calendar (one row per night, one year ahead). Booking is the hard consistency piece: transactionally flip all N target-nights from available → held; 15-minute hold while payment processes; on payment success, held → booked. Pricing includes host base + overrides + smart-pricing suggestion + fees + taxes computed at booking time. Trust and safety runs risk models per booking.

56Reddit

Big design problems: ranking (posts and comments), threaded comments at scale, vote fuzzing to defeat bots. Hot ranking: log10(max(|score|,1)) × sign(score) + seconds/45000 — logarithmic diminishing returns on votes, linear time decay. Best comment ranking uses the Wilson-score lower bound — resistant to small-sample noise. Comments stored with (post_id, parent_id, comment_id, score, created_at); hot subreddits get pre-computed feeds; smaller ones compute on demand.

57Google Search

Pipeline: crawl → index → serve → rank → snippet. Inverted index sharded by document — one query touches every shard in parallel; a slow shard is skipped past its budget. Ranking blends hundreds of signals (PageRank, BERT/MUM semantic match, click model, freshness, site quality, personalization). Tail latency control via backup requests (send to a replica after p50 hits). Total query budget ~200ms — the whole world of distributed systems compressed into one round trip.

58Amazon S3

"Eleven nines" durability via erasure coding, not plain replication. An object is split into K data chunks + M parity chunks (Reed-Solomon); any K of K+M recovers the object. ~1.4× storage overhead vs 3× for triple replication. Chunks are spread across racks and AZs. The service is three layers: request routers (auth, quota, index lookup) → a keymap service (bucket + key prefix → location) → storage nodes. Strong read-after-write consistency since 2020 via a coordinator-based read protocol.

59Kafka Internals

What makes it fast: append-only log on disk (sequential I/O is ~100× random on SSDs), zero-copy from page cache to socket via sendfile, batching + compression at the producer, and per-topic partitioning for parallel scale. Each partition has a leader and followers; ISR (in-sync replicas) determines who's eligible to become leader. Consumers track their own offsets in the __consumer_offsets topic. KRaft (Kafka's Raft) has replaced ZooKeeper for metadata in modern deployments.

60LLM Serving

Each new token = one full forward pass through billions of parameters. Prefill (first pass over the prompt) is compute-bound; decode (one token at a time) is memory-bandwidth-bound — different optimizations. Batching across requests keeps GPUs busy. KV cache stores past attention states so we don't re-compute them per token. Prompt-prefix caching helps when many users share a long system prompt. Speculative decoding: small draft model proposes tokens; big model verifies in parallel. Quantization (int8 / int4) cuts memory and cost.

61Stock Exchange

The order book is a pair of sorted structures (buys descending, sells ascending). Matching is price-time priority: best price wins, ties broken by arrival. Matching engines are single-threaded per symbol for determinism; symbols scale horizontally. p99 latency in tens of microseconds. Latency tricks used by HFT: kernel-bypass NICs (DPDK, Solarflare), FPGA matching, co-located racks inside the exchange, microwave links between exchange cities. Trades match in T+0; clearing and custody settle T+1 or T+2.

62Google Docs (Real-Time Collaboration)

Two concurrent inserts at the same position must converge to the same final state for every viewer. Operational Transformation: each edit is an operation; when two are concurrent, the server transforms each against the other so intent is preserved (Alice's insert at 0 and Bob's insert at 5 both land where they should). Server is the single source of truth for op order; snapshots the doc periodically so replay isn't from genesis. Presence + cursors broadcast on WebSocket. Offline edits are buffered locally and transformed on reconnect.

63Tinder

~7.5B swipes/day. Data model: swipes (write log), matches (derived), profiles. Candidate generation: geohash bucketing + ES filter + a personalization model re-ranking the top 100. Match detection on a right-swipe: insert the row, then check if the target already right-swiped me — that reverse lookup must be O(1) (Redis set right_swipes_by:{target_id}). Recent activity boosts ranking — Tinder wants matches to chat now.

64Nginx Internals

Event-driven architecture. A master process manages a handful of workers (usually one per CPU core). Each worker is single-threaded and uses epoll/kqueue to multiplex thousands of sockets. Memory per connection is tiny (~2KB), which is why one box handles 100K+ concurrent connections where Apache's thread-per-request model would collapse. Requests flow through configurable phases (post_read → rewrite → access → content → log). In reverse-proxy mode, Nginx maintains connection pools to upstreams and buffers slow clients so backends never see them.

65Real-Time Leaderboard

Redis sorted sets. ZADD leaderboard:daily 1500 user:42; ZRANGE leaderboard:daily 0 99 REV for the top 100; ZREVRANK for a user's rank. Insert and range are O(log N). For tens of millions of entries, shard by user_id bucket, keep local top-K per shard, and merge — accepting eventually-correct top-K, which is fine for display. Multiple leaderboards (daily, weekly, all-time, per-region) as separate keys. Persist via Redis AOF and periodic snapshots.

66Apple AirTags

An AirTag has no GPS or cellular. It broadcasts a rotating BLE beacon. Any nearby iPhone hears it, encrypts (its own GPS location + time) with the AirTag's rotating public key, and uploads to Apple. The owner's phone fetches the encrypted blobs for their AirTag's keys and decrypts locally — only the owner holds the private key. Apple servers store blobs they can't read. The helper iPhones don't know whose AirTag they reported. UWB takes over for cm-accurate direction when the owner is close. Anti-stalking: unknown AirTag alerts and audible sounds after prolonged separation.

67Vitess — YouTube's MySQL at Scale

YouTube's core metadata still runs on MySQL, sharded via Vitess (now CNCF open source). Applications write regular SQL to VTGate, a stateless proxy that parses the query and routes it to the right shard. VTTablet sits in front of each MySQL. Vitess provides transparent sharding, online resharding (split a shard live without downtime), connection pooling (thousands of app connections → handful of MySQL connections), query rewriting, and replication topology management. Take-away: sharding doesn't require abandoning SQL.
